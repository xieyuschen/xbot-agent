# xbot

> Go AI Agent framework with message bus + plugin architecture. Supports Feishu/QQ/CLI/Web channels, tool calling, pluggable memory, skills, subagents, MCP integration.

## Quick Reference

- Entry points: `cmd/xbot-cli/` (CLI), `cmd/runner/` (remote sandbox), `cmd/xbot/` (server)
- Build: `go build ./...` | Test: `go test ./...` | Lint: `golangci-lint run ./...`
- Config: `~/.xbot/config.json`, env var overrides
- Subscriptions: `~/.xbot/config.json` (CLI) or DB `user_llm_subscriptions` (Server) — the single source of truth for LLM config
- Pre-commit: gofmt → golangci-lint → go build → go test

## Gotchas — MUST READ Before Any Code Change

### Concurrency
- **Never `defer` semaphore release inside a loop.** Deadlock when iterations exceed capacity. Release immediately after Generate completes.
- Non-blocking channel sends: always use `select` with `ctx.Done()` to prevent blocking on full channels during shutdown.
- **User-scoped semaphores must not be hardcoded to capacity 1 when one sender can own multiple independent chats/sessions (for example remote CLI windows authenticated as `admin`).** Size them from configured concurrency or key them by session, otherwise different windows will block each other and look like a leaked semaphore.
- **`SetMaxConcurrency` must clear `userSemaphores` cache.** The global semaphore is rebuilt with the new capacity, but `getUserSemaphore` caches per-user channels in a `sync.Map` via `LoadOrStore`. Without `Clear()`, users with custom LLM keep using the cached semaphore with the OLD capacity forever. Symptom: setting max_concurrency to 100 has no visible effect.
- **`cancelChildSessions` must only cancel sessions with matching `parentKey`.** The old code called `cancelCurrent()` on ALL interactive sessions inside the `Range` loop before checking `parentKey`. This killed sibling/peer background agents when any single agent was unloaded or panicked — all N peer agents die simultaneously at whatever time the first one finishes. The `parentKey` check must happen BEFORE `cancelCurrent()`.
- **SubAgents must never outlive their direct creator — ALL completion paths need `cancelChildSessions`.** Foreground SubAgents wrap their own `fgRunCtx` (with `bgSessionCtxKey`+`bgParentKey`) so children derive context from the creator, not a grandparent. Send path `runCtx` also has these markers. Every natural completion path (foreground, bg, send) calls `runCancel()` + `cancelChildSessions(key)`. Without the context wrapper, a foreground B creating bg C would have C's context chain skip B entirely → B finishes, C keeps running.

### Subscription & Settings
- **`agent/setting_runtime.go` is the SINGLE source of truth for runtime setting handlers.** Both CLI and server use it. Never create a second handler registry — it will silently diverge.
- **`backendSubscriptionManager` is the SINGLE SubscriptionManager implementation.** No more local/remote/config variants. All subscription operations go through Backend interface → Transport (local or remote).
- **Never use `UpdateSubscription` for PerModelConfig changes.** Use `UpdatePerModelConfig(subID, model, pmc)` — it only touches PerModelConfigs, never touches credentials. The old List→modify→Update pattern reads masked keys from the API, then writes them back, destroying real credentials.
- **`serverapp/rpc_table.go:updateSubscription` starts from EXISTING subscription, only overlays non-masked fields.** Client sends masked keys (****) — the handler preserves real credentials from DB.
- **`max_context_tokens` is per-model, NOT per-subscription.** Stored in `PerModelConfigs[model].MaxContext`, NOT in user_settings DB or config.Agent.MaxContextTokens. `/settings` writes only the current session's model via `UpdatePerModelConfig`, preserving existing `MaxOutputTokens`. The old `SetMaxContextTokens` / `set_max_context_tokens` RPC / `perChatMaxCtx` map / `max_context_tokens` setting handler have all been REMOVED — settings writes go directly to DB via `UpdatePerModelConfig`, and `GetEffectiveMaxContext` reads from DB via `resolveSubContextFor`. Must NOT pass `max_context_tokens` to `ApplySettings` — there is no handler for it anymore.
- **`thinking_mode` is a GLOBAL user setting, NOT subscription-scoped ("订阅是订阅，模型是模型").** Stored in `user_settings` under canonical channel `channel.ThinkingModeChannel == "cli"` (constant lives in the `channel` package to avoid the `channel/cli ↔ agent` import cycle). Surfaced via the main chat status-bar indicator (`🧠 auto|on|off [Ctrl+M]`, `renderReadyStatus` + `thinkingZoneXStart/End` click zone) and `Ctrl+M` toggle (`toggleThinkingMode` cycles `""→enabled→disabled→""`). `ResolveLLM` reads it via `f.userThinkingMode(senderID)` — priority per-model override (`pmc.thinkingMode`) → global user setting → `""`. `sub.ThinkingMode` is NO LONGER read (kept only as an internal fallback field). The `thinking_mode` `SettingHandlerRegistry` handler calls `f.InvalidateSender(senderID)` (no-op now — no in-memory cache to clear; DB is read fresh every turn). `cmd/xbot-cli/main.go` `ApplySettings` must NOT treat `thinking_mode` as an llm field (`updateActiveSubscription` no longer writes `sub.ThinkingMode`) — thinking flows only through `ApplyRuntimeSettings` → handler → `InvalidateSender`. Startup seeds the global user setting once (`serverapp`: `cfg.LLM.ThinkingMode` → default-sub `ThinkingMode` → `""`). Subscription add/edit/display panels are credentials-only (name/provider/base_url/api_key); `setLLMCmdForSub` shows only credential params.
- **`refreshRemoteValuesCache` MUST extract `max_context_tokens` from `PerModelConfigs[sub.Model].MaxContext`.** The function populates `valuesCache` which feeds `GetCurrentValues()` → `resolveMaxContext()` → context bar display. Without this extraction, the cache falls through to the `config.DefaultMaxContextTokens` (200000) fallback, causing the context bar to show wrong max context after settings save. The extraction must mirror the settings panel read path (`cli_settings.go:79-93`): PerModelConfigs[model].MaxContext → sub.MaxContext → config fallback. Also must NOT overwrite existing DB values from `GetSettings` (config tool wrote them via `SetSetting`).
- **`SwitchModel` with chatID must NOT call `SetModel` on the subscription.** Per-session model switches (with chatID) only create per-chat cache entries. Calling `SetModel` contaminates all sessions sharing the same subscription. Only user-level switches (no chatID) update the subscription.
- **`PerModelConfigs` is a read-side projection populated by the storage layer (v42).** `subscription_models` is the sole source for per-model config; the `per_model_configs` JSON column was dropped. `LLMSubscriptionService.List`/`Get`/`GetDefault` call `loadPerModelConfigs` (which reads `subscription_models`) AFTER closing the rows cursor — a nested query inside `rows.Next()` deadlocks the single-connection SQLite pool. `Add` upserts the incoming map into the table; `UpdatePerModelConfigs` does delete+reinsert. The old `mergeSubscriptionModels` in rpc_table was deleted (no longer needed). Client-side `ResolveEffectiveMaxContext`/`sub.PerModelConfigs` now see table values without an explicit merge step.
- **`IsDefault`/`Active` are read-side projections derived from `user_default_model` (v43).** The `is_default` column was dropped. `GetDefault` reads `user_default_model` → `Get(subID)` and marks `IsDefault=true`; `List`/`ListAll` mark via `markDefaultsFor`/`markDefaultsAll`. `SetDefault(id)` upserts `user_default_model`. `Add`/`Update` seed/sync `user_default_model` when `IsDefault=true`. `Get` does NOT populate `IsDefault` (stays false) — callers needing default-status use `GetDefault`/`List` or check `user_default_model` directly (e.g. `updateSubscription` RPC re-switches the user-level entry only if `GetUserDefaultModel` matches the edited sub).
- **System subscription is the single source for the global default LLM (v44).** The global default LLM is a shared DB row (`user_llm_subscriptions` with `sender_id="__system__"`, `is_system=1`, `id="system"`), reconciled from `cfg.LLM`/env every boot by `serverapp.reconcileSystemSubscription`. Do NOT rebuild `defaultLLM` from `cfg.LLM` directly or via `SetDefaults` — use `LLMFactory.SetSystemLLM(client, model)` (which does NOT clear per-user caches). `GetDefault(senderID)` falls back to the system subscription when a user has no `user_default_model`. `List`/`ListAll` inject the system row for every user. The system subscription is read-only: `Remove`/`SetSubscriptionEnabled`/`Rename`/`Update` refuse `is_system=1` rows (storage guards + UI `subIsSystem` + 🔒 badge). `saveServerConfig`/`saveCLIConfig` must NOT write `cfg.LLM` credentials back to config.json — `cfg.LLM` may hold decrypted values refreshed from DB, and the DB system subscription is the single source (config.json keeps existing creds only as a boot seed via `SaveToFile` deep merge). Picker system models now come from the system subscription row (SubName="system"), not from `defaultLLM.ListModels()` pseudo-entries.
- **`handleSwitchLLMDoneMsg` must NOT pass `done.maxCtx` as `state.MaxContextTokens`.** Those values come from `resolveSubMaxContext` using the subscription's DEFAULT model. Setting them poisons `ResolveEffectiveMaxContext` priority-1 shortcut, bypassing per-model lookup. Let `applySessionLLMState` resolve dynamically from `PerModelConfigs`.
- **`scheduleSessionLLMRestore` must use `m.cachedModelName` (per-session model from `refreshCachedModelName`), NOT `target.Model` (subscription default).** The old code always used `target.Model`, so per-session model choices (Ctrl+N switch) were lost on restart — the restored LLM used the subscription's default model. `handleSwitchLLMDoneMsg` must also call `SwitchModel(senderID, done.subModel, chatID)` AFTER `SetDefault(subID, chatID)`, because `SetDefault`→`SetSessionLLM` creates the per-chat entry with `sub.Model` (subscription default), overriding the per-session model. The `SwitchModel` call corrects the model field and re-persists to the tenants table.
- **`OnModelsLoaded` callback runs in `NewOpenAILLM`'s async goroutine** — must be concurrency-safe.
- **Tier fallback**: unconfigured tier → vanguard→balance→swift chain. Empty tier must NOT return default client with wrong model.
- **`createClientFromSub` uses sub's credentials with a *different* model** — verify target model is served by that endpoint.
- **Model-first redesign (v39): `ResolveLLM` is the authoritative agent-loop entry.** It reads **DB directly** every call (no in-memory cache): ProxyLLM → tenants table → `user_default_model` → legacy `GetLLM` fallback. The old `sessionMemo`/`entries`/`hasCustomLLMCache`/`perChatMaxCtx` in-memory caches have been REMOVED — `Invalidate`/`InvalidateSender`/`InvalidateSession` are now no-ops (DB is authoritative), `InvalidateAll` only clears `clientCache`. `GetLLMForChat` reads per-session (subID, model) from tenants table, falls back to `GetLLM` (global default). `SetSessionLLM` writes to tenants table via `tenantSvc.SetTenantSubscription`. New LLM features must build on `ResolveLLM`/`SelectModel`, not the removed caches. See `docs/agent/subscription.md` "Model-First Redesign".
- **Cross-subscription model switch 404: `switch_model` per-session must resolve the owner subscription, not use `GetDefault()`.** The old code paired `(defaultSub.ID, anyModel)` into tenants — switching to a model from another subscription built a client from the default sub's baseURL/apiKey but requested the other model name → 404 "model not supported by any configured account". Fix: `ResolveSubscriptionForModel(senderID, model)` finds the enabled subscription that provides the model, then `SelectModel(owner.ID, model, chatID)`. Fallback to default-sub only when no owner resolves.
- **`clientCache` (keyed by `(subID, apiType)`) must be invalidated on every subscription CRUD.** `add/remove/update/set_default/set_subscription_model/update_per_model_config` RPCs and `LLMAdd/Remove/SetDefault/Update` callbacks must call `LLMFactory().InvalidateSubscription(subID)`, or `ResolveLLM` reuses a client built from stale credentials.
- **Disabled models (`subscription_models.enabled=0`) are excluded from `ListAllModelsForUser` and rejected by `SelectModel`/`ResolveSubscriptionForModel`.** `protocol.PerModelConfig.Enabled` is a read-side projection (populated by `mergeSubscriptionModels`); writes go through the `set_model_enabled` RPC, never through `UpdatePerModelConfig`.
- **v39 migration coexists with master's v38 (`tenants.runner_id`).** model-first originally used v38; after rebase it was bumped to v39 to avoid the schema-version collision. `schemaVersion=40` (v40 added `user_llm_subscriptions.enabled`). A live DB that reached v38 via the pre-rebase model-pool binary lacks `runner_id` and needs a manual `ALTER TABLE tenants ADD COLUMN runner_id TEXT DEFAULT ''` before running the new binary (dev-machine-only; not in migration code).
- **Subscription-level enabled (v40) must be skipped in three places.** `user_llm_subscriptions.enabled=0` must be skipped by `ListAllModelsForUser` (both loops), `ResolveSubscriptionForModel` (`find` closure), and rejected by `SelectModel` — or disabling a subscription is cosmetic. `SetSubscriptionEnabled` must call `InvalidateSubscription(subID)`. `protocol.Subscription.Enabled` is a read-side projection from `user_llm_subscriptions.enabled` (via `subToChannel`/`LLMGetSubscription`); writes go through the `set_subscription_enabled` RPC.
- **No subscription switching in the UI; one unified LLM panel.** `cli_panel_quickswitch.go` is a SINGLE panel (opened by `Ctrl+N` / click status-bar model name / palette "Models & Subscriptions") that combines subscriptions + models + add actions in one flat `[]qsRow` list (`qsSection`/`qsSub`/`qsModel`/`qsAddSub`/`qsAddModel`). There is NO separate "subscription" panel and NO `Ctrl+P`/`Ctrl+N` anymore. Subscription rows are management-only (add/disable/delete); model rows switch cross-subscription. The old `SwitchLLM` async + `cliSwitchLLMDoneMsg` path is retained only for startup restore (`scheduleSessionLLMRestore`). Key scheme (command mode, no filter active): `↑↓` navigate (skips section rows), `Enter` = row action (sub→toggle enabled, model→switch, add row→open add panel), `E` edit current row (sub creds / model params), `D` disable-or-delete (model→toggle enabled, sub→delete), `N` add model (prefills owner from cursor row), `S` toggle `quickSwitchShowAll`, `/` enter filter mode, `Esc` close. **`/` toggles filter mode** so command letters (`e`/`d`/`n`/`s`) never collide with typing — this replaced the old Ctrl-combo commands (`Ctrl+E`/`Ctrl+A`/`Ctrl+T`) which conflicted with global `Ctrl+E` (fold) and `Ctrl+T` (sessions). `applyQuickSwitch` reads `quickSwitchRows[cursor]` and **rejects `Status=="disabled"`** models (keeps panel open, hints press `E` to enable); `normal` AND `offline` are both selectable. Both switch paths use `applyModelSwitch`, which after `SwitchModel` re-reads the owner subscription via `GetSessionSubscription` to fix `activeSubID`/context limits (tenants table is source of truth in both local and remote modes). Status bar shows `模型名(订阅名)` from `cachedSubName` (refreshed by `refreshCachedSubName` on `activeSubID` changes — NOT per-frame). `handleQuickSwitchKey` intercepts Esc/Up/Down/Enter over `quickSwitchRows` BEFORE other handling; in filter mode letters feed `quickSwitchFilterInput` and `rebuildLLMRows` re-filters. **Panel scrolling**: `quickSwitchScrollY` tracks vertical scroll offset; `ensureQuickSwitchCursorVisible` keeps cursor in view; `viewQuickSwitch` slices rows to `height - overhead` visible lines with a `── N-M / Total ──` scroll indicator. **Contextual hints**: `buildQuickSwitchHint()` renders row-type-specific operation hints (sub/model/add/section) in `PanelHint` style + common keys in `TextMutedSt`. **Model selection**: `✓` mark and `cursorToActiveLLMRow` match both `Model` AND `SubID == activeSubID` — same model name from different subscriptions is NOT a match. **Panel colors**: non-cursor rows use `TextSecondarySt` (not `TextMutedSt`); section headers use `PanelDivider`; disabled/offline models use `TextMutedSt`; cursor row uses `Accent`. **Thinking indicator**: `thinkingModeLabel()` returns ASCII `think`/`think+`/`think-` (no emoji, no `[Ctrl+M]` suffix); `think+` uses `Accent` style, `think-` uses `TextMutedSt`.
- **Model list is DB-driven; each entry has a Status (normal/offline/disabled).** `listModelEntriesCore` lists the **union** of `sub.CachedModels` + `sub.Model` + `subscription_models` rows — i.e. everything recorded in the DB, not just fetched. `Status` per (SubID,Model): `disabled` if `subscription_models.enabled=0`; else `normal` if in `CachedModels` or is `sub.Model`; else `offline` (has a record but not fetched — manually added, still selectable). `ListAllModelEntriesForUser` (includeDisabled=true, picker) emits all three; `ListAllModelsForUser` (includeDisabled=false, tier selectors) emits `normal`+`offline` only. Owner attribution picks the best sub by status rank (normal>offline>disabled). The `CachedModels` snapshot is populated by the `OnModelsLoaded` callback — which the model-first refactor had **dropped** (must be re-wired via `makeOnModelsLoaded(subID)` in `createClientFromSub` + both entry-construction paths; the callback nil-checks `Get(subID)` before `UpdateCachedModels` since the latter nil-derefs on missing sub). Opening the panel (`openLLMPanel`) kicks off `RefreshModelEntries()` → RPC `refresh_model_entries` → `LLMFactory.RefreshModelEntriesForUser`: parallel (cap 8, 8s/sub, soft-fail) `/models` fetch per enabled subscription, persist via `OnModelsLoaded`, return fresh entries (a fetched model flips `offline`→`normal`). CLI renders the DB snapshot immediately, shows `↻ 刷新模型列表…`, and rebuilds rows on `cliModelEntriesRefreshedMsg` (handled at top of `Update`). `rebuildLLMRows` clamps the cursor (never resets) so a background refresh doesn't yank the user's cursor; callers set it explicitly (`cursorToActiveLLMRow` on open, `0` while typing). `backendModelLister.EnsureModelsLoaded` is a no-op — refresh is driven by `RefreshModelEntries`, don't rely on `EnsureModelsLoaded`. The three entry points (Ctrl+N / click model name / palette Enter) must drain `m.pendingCmds` so the refresh cmd actually runs. **`/models` per-provider differences are expected** (different providers expose different model lists); the panel shows all DB records regardless, `N` lets users register models the provider didn't list (appears as `offline`), and `isNoiseModel` hides chat-unusable noise (image/realtime/whisper/tts/audio/embed/moderation/dated-snapshots) unless `S` shows all. **Per-model params are edited ONLY on the model row via `E`** (`openEditModelPanel` → `UpdatePerModelConfig`/`SetModelEnabled`); the subscription edit panel (`openEditSubscriptionPanel`) edits credentials/defaults only and NO longer carries per-model toggles — that was redundant with the model row editor.

### Context Management & Token Estimation
- **`maybeCompress` uses ONLY API-returned `prompt_tokens` via `TokenTracker.GetPromptTokens()`.** No local estimation (tiktoken/CountMessagesTokens) in the compression decision path. The `no_data` source (no API data yet) skips all compress/masking checks. Tests must set `cfg.LastPromptTokens` to simulate a previous Run. **`maybeCompress` does NOT add `completionTokens` to its check — `shouldCompact(totalTokens, promptBudget, threshold)` compares `prompt_tokens` against `(maxContextTokens - maxOutputTokens) * threshold`, pure prompt tokens only.** Both the context bar and maybeCompress use the same `tokenTracker.GetPromptTokens()` source — they never diverge.
- **`ResetAfterCompress()` takes no arguments — it zeros ALL tracker fields** (promptTokens, completionTokens, hadLLMCall). Any non-zero value causes `maybeCompress` to re-trigger immediately, creating infinite compression loops. The tracker returns "no_data" until the next real LLM API call.
- **`RecordLLMCall(prompt, completion)` takes 2 args only.** No msgCount — the old delta estimation via `CountMessagesTokens` has been removed. Offload handles large tool results; a single iteration cannot add enough tokens to justify local estimation.
- **`RecordLLMCall` must NOT be called when `response.Usage.PromptTokens == 0`.** When a stream is cancelled mid-flight (Ctrl+C), `CollectStreamWithCallback` returns a non-nil response with zero Usage (usage events arrive at stream end). Calling `RecordLLMCall(0, 0)` overwrites the tracker's correct value from a previous iteration. This causes `SaveState` to skip (`promptTokens <= 0`), but `SaveTokenState(0, 0)` and `SaveContextTokens(0)` in `callLLM` still execute unconditionally, overwriting DB with 0. Next turn restores 0 → context bar empty → first callLLM corrects to real value → immediate compression. Fix: guard all three save paths (`RecordLLMCall`, `SaveContextTokens`, `SaveTokenState`) with `response.Usage.PromptTokens > 0`.
- **`buildToolContextExtras` uses `TenantSession.MemoryService()` for `MemorySvc`/`TenantID`, NOT `LettaMemory` type assertion.** These are tenant-level fields that work for all memory providers. Only LettaMemory-specific fields (CoreMemory, ArchivalMemory, ToolIndexer) stay inside the type assertion.
- **`ObservationMaskStore` and `OffloadStore` both persist to disk.** Mask uses `~/.xbot/mask/{tenantID}/{id}.json`, Offload uses `~/.xbot/offload_store/{session}/{id}.json`. `Recall` falls back to disk on memory miss. Both cleaned on compress and `/clear`.
- **`PersistenceBridge` manages the persistence watermark (`lastPersistedCount`), not inline fields.** All compression paths use `ApplyCompress` pipeline which calls `PersistenceBridge.RewriteAfterCompress()` to atomically clear+rewrite+update watermark. The invariant `LastPersistedCount <= len(messages)` is verified by `ValidateInvariants()` at debug level after every LLM call, compression, and persistence operation.
- **Shell binary output silently bypasses offload: `summarizeShell` splits by `\n` and keeps last 5 lines.** Binary data (e.g. `cat libmujoco.so`) has very few newlines, so "last 5 lines" ≈ entire content. The offload summary then contains nearly all original binary data, causing context explosion (39k→2.2M tokens observed). `summarizeShell` now applies `maxLineRunes=500` truncation (same as `summarizeRead`) on output lines. But any new summary generator must handle the few-lines-but-megabytes-per-line case.
- **`SanitizeMessages()` is the final validation gate before sending to LLM.** Five passes: (1) strip assistant messages with empty content AND no tool_calls — rejected by OpenAI/Anthropic; (2) strip tool_calls with invalid/malformed JSON arguments — caused by cancelled stream with partially-generated tool arguments, DeepSeek rejects with 400; (3) strip trailing unpaired tool_calls from cancelled runs; (4) ensure every tool_call_id has a matching tool response message — strip orphaned tool_calls, re-run Pass 1 to clean up emptied assistant messages; (5) strip orphaned tool messages whose tool_call_id doesn't appear in any assistant's tool_calls — Pass 2 and Pass 4 can strip tool_calls leaving orphaned tool messages, DeepSeek rejects with "Messages with role 'tool' must be a response to a preceding message with 'tool_calls'". Called in `buildPrompt` and `engine.go`.
- **`ConvertMessagesToHistory` infers tool status from tool messages when Detail is absent.** When the server crashes mid-turn, persisted assistant messages have ToolCalls but NO Detail (Detail is only set at end-of-turn in `agent_process.go`). The old code hard-coded `Status: "done"` for all tools in this fallback path — failed tools rendered as successful after TUI restart. Fix: pre-scan tool messages to build a `toolCallID → content` map; if content starts with `"Error:"` (the prefix `engine_run_tools.go:updateToolResultLine` adds for `IsError` results), status becomes `"error"`, otherwise `"done"`. `cursorToActiveLLMRow` and the `✓` mark in `viewQuickSwitch` match both `Model` AND `SubID == activeSubID` — same model name from different subscriptions is NOT a match.
- **`CollectStreamWithCallback` must return partial content on `ctx.Done()`, not nil.** When user cancels streaming mid-reasoning, returning nil loses accumulated `reasoning_content` and partial `tool_calls`. The partial tool_calls may have broken JSON arguments, which then pass through `SanitizeMessages` Pass 2 for cleanup. Returning nil also prevents the engine from properly recording the cancellation state.
- **`GenerateStreamAndCollect` must NOT use `perAttemptCtx` for streaming.** A per-attempt deadline binds to the underlying HTTP connection via ctx. The deadline kills active streams mid-generation even when chunks are still flowing (e.g., DeepSeek long reasoning > 120s). Stream timeout is managed by `CollectStreamWithCallback`'s idle timeout (120s without any chunk) — timer resets on every received chunk, so actively-streaming responses never time out.
- **MCP stdio `exec.Command` uses process PATH, not `cmd.Env`.** Go's `exec.Command("npx", ...)` resolves the executable using the *process* PATH (`os.Getenv("PATH")`), ignoring `cmd.Env`. When xbot runs as a service with minimal PATH, tools like npx/nvm are invisible. `ConnectStdioServer` now calls `resolveCommand()` to pre-resolve the absolute path using the login shell PATH from `getLoginShellEnv()`. Also, `getLoginShellEnv` uses `bash -i -l` (interactive login shell) because `bash -l` alone skips `.bashrc` where nvm initializes.

- **SQLite `datetime()` comparison with RFC3339 strings is a silent trap.** `created_at` stores `2026-05-24T14:00:00+08:00` (RFC3339 with timezone). `datetime(?, '-2 seconds')` returns `2026-05-24 05:59:58` (UTC, no timezone). Raw string comparison: `'T'` (0x54) > `' '` (0x20) at position 10, so `created_at > datetime(?)` is **always TRUE**, breaking any time-window dedup/filter. Always wrap both sides in `datetime()`: `datetime(created_at) > datetime(?, '-2 seconds')`.

### Cron Scheduler
- **`cleanupExpiredJobs` must NOT delete `delay_seconds` one-shot jobs.** `delay_seconds` represents relative time ("N seconds from creation"). If the job expired during downtime, deleting it silently drops the user's scheduled reminder. Keep expired `delay_seconds` jobs so `checkAndFire` triggers them on the first tick. Only `at`-based one-shot jobs (absolute time point) should be removed when expired — the scheduled moment has passed.
- **`every_seconds` next_run must be based on `job.NextRun`, NOT `now`.** The old code used `nextRun = now.Add(interval)`, where `now` is the actual fire time (delayed by 0–5s due to the 5s ticker). Each fire accumulates drift, causing the schedule to slide later and later. Fix: `nextRun = job.NextRun.Add(interval)` anchored to the planned schedule, with a forward-advance loop for missed intervals.
- **Recurring jobs skip missed executions on restart (by design).** `cleanupExpiredJobs` recalculates `next_run` from `now` for recurring jobs. This matches traditional cron semantics — no catch-up execution.

### Startup

### CLI Deterministic Rendering
- **Every assistant/tool_summary message is keyed by `agentTurnID` + `role`.** `upsertMessageByTurn(turnID, role, msg)` finds existing entries and updates in-place instead of appending duplicates. This prevents duplicate messages when PhaseDone and cliOutboundMsg arrive out of order.
- **`upsertMessageByTurn` now also purges zombie duplicates.** After updating a message, it scans for OTHER messages with the same (turnID, role) pair and removes them. Without this, fallback paths (e.g. isPartial fallback creating a second streaming message) could leave zombies that render as duplicates.
- **`dedupMessagesGuard()` runs at the TOP of `updateViewportContent()`.** This is the algorithmic guarantee layer: it scans `m.messages` for any (turnID, role) duplicates (turnID > 0) and purges earlier occurrences before rendering. Uses O(n) map-based identity detection — NOT string matching. Even if a race condition or unguarded append path creates a duplicate, it is silently purged here before reaching the viewport.
- **`endAgentTurn` clears `streamingMsgIdx = -1`.** Without this, a delayed complete message from the previous turn can overwrite the new turn's streaming message via the stale index. The non-streaming path in `handleAgentMessage` also checks `m.messages[m.streamingMsgIdx].turnID == turnID` before overwriting.
- **`isPartial` fallback checks `findMessageByTurn` before creating new message.** When `streamingMsgIdx` is stale (cleared by endAgentTurn), the fallback now searches for an existing message with the same turnID+role and reuses it, instead of appending a duplicate.
- **Cache generation counter (`histGen`/`allLinesGen`) prevents stale `allLines` reuse.** Every site that modifies `histLines` calls `bumpHistGen()`. The tick fast path only reuses `allLines` when `histGen == allLinesGen`. This replaces the old length-only comparison (`histLen == allLinesHistLen`) which could not detect content changes when line count coincidentally matched — causing stale cache + new content to render together as duplicates.
- **`turnDoneFlags` tracks per-turn completion state**: `doneProcessed` (PhaseDone created tool_summary) and `replyReceived` (handleAgentMessage appended assistant reply). `handleAgentMessage` checks `doneProcessed` to skip redundant tool_summary creation. **Cleared on session switch** in `postRestoreSessionSetup` to prevent stale flags from causing premature queue flush.
- **Queue flush requires `replyReceived` or `doneProcessed+turnCancelled`.** The old heuristic (`!typing` on next tick) could flush before the assistant reply arrived. Now the tick handler waits for `replyReceived=true` on the completed turn. A 2s timeout fallback prevents permanent queue stalls when replies are lost.
- **`pendingToolSummary` is no longer used for PhaseDone→handleAgentMessage handoff.** The upsert-by-turn mechanism replaces it — `handleProgressDone` creates the tool_summary via upsert, and `handleAgentMessage` either finds it already there (doneProcessed) or creates it from local iteration history.
- **Cancelled turns must bake iterations into the streaming message BEFORE `endAgentTurn` clears `iterationHistory`.** The cancel path in `handleProgressDone` copies `iterationHistory` into `streamingMsg.iterations` before calling `endAgentTurn()`. Without this, all tool tags and reasoning rendered inline during the cancelled turn vanish after Ctrl+C. The cancel ack in `handleAgentMessage` also finalizes (not removes) the streaming message when it has iteration data — even if `content == ""` (e.g. agent was mid-tool with no text yet). Only truly empty messages with zero iterations are removed.
- **Turn completion uses `rerenderCachedMessage` instead of `fullRebuild`.** When the complete reply arrives in `handleAgentMessage`, instead of setting `rc.valid = false` (which forces O(N) `fullRebuild` → flicker), call `rerenderCachedMessage(completedMsgIdx)` which truncates the render cache back to that message and re-renders only it via `appendNewMessagesToCache` — O(1 message) instead of O(N). The cancel ack path uses the same mechanism via `cancelledIdx`.
- **`endAgentTurn` must NOT clear `streamingMsgIdx`.** Clearing it causes the subsequent `relayoutViewport→updateViewportContent` to call `appendNewMessagesToCache` and cache the streaming message with incomplete content (reply hasn't arrived yet). This forces a second viewport update when `rerenderCachedMessage` runs, causing double-flicker. Instead, keep `streamingMsgIdx` so `updateViewportContent` uses `updateStreamingOnly` (streaming path) — the message is NOT cached prematurely. `handleAgentMessage` will set `streamingMsgIdx = -1` and call `rerenderCachedMessage` for a single clean transition.
- **`snapshotIterationChange` falls back to `ReasoningStreamContent`.** When an iteration changes and the previous iteration's reasoning hasn't been finalized (structured `Reasoning` is empty, `reasoningByIter` missed it), the snapshot would lose the reasoning block — causing it to briefly disappear during streaming. The fallback (`prev.ReasoningStreamContent`) captures the live-streamed reasoning, matching the pattern already used in `handleProgressDone`'s cancel path.
- **Server-side cancel barrier via `ctx.Done()`** prevents stale progress events from arriving after Ctrl+C. `buildMainRunConfig` wraps every `ProgressEventHandler` with `select { case <-ctx.Done(): return; default: }`. When `/cancel` triggers `reqCancel()`, the ctx is cancelled, and all subsequent progress event handler calls return immediately — no stale progress/PhaseDone events leak through to trigger re-renders. No extra fields needed: the ctx is the natural cancellation mechanism already wired through `chatProcessLoop → processMessage → buildMainRunConfig`.
- **`handleHistoryLoad` now deduplicates using identity keys.** Messages are keyed by `role + ":" + turnID + ":" + content` and duplicates are skipped during append. Prevents double-rendering when history reload races with in-flight messages.
- **`compressionReloading` and `turnDoneFlags` are cleared in `postRestoreSessionSetup`.** Without clearing, a stale `compressionReloading=true` from the previous session permanently blocks auto-start in the new session, and stale `turnDoneFlags` can cause premature queue flush.
- **`HistoryCompacted` handler MUST immediately create a streaming message and clear `pendingUserMsg`.** The old code set `compReloading=true` to block auto-start until the async reload completed, then relied on `handleHistoryReload` to recreate the streaming message. But `reloadMessagesFromSession` dropped the message on full `asyncCh` (silent `default`), making `compReloading` permanently true — no streaming message was ever created, freezing the TUI (busy but no live updates). Now the streaming message is created immediately, progress events render via `updateStreamingOnly` before the reload even arrives. `pendingUserMsg` is cleared to prevent duplicates: the reload fetches the user message from DB with system-guide text prepended, which doesn't match `pendingUserMsg.content` (raw input), causing `handleHistoryReload` to append it again.
- **`handleHistoryReload` MUST preserve the streaming message for ALL reloads, not just non-forceFullRebuild.** The old `!msg.forceFullRebuild` check meant forceFullRebuild (from compression) replaced messages WITHOUT preserving the streaming message. Combined with `compReloading` blocking auto-start, this left no streaming message at all. Now always preserves `streamingMsg` when `m.typing && m.streamingMsgIdx >= 0`.
- **`reloadMessagesFromSession` MUST use blocking send with timeout, NOT silent `default` drop.** The old `select { case ... asyncCh <- msg: default: }` silently dropped the reload when asyncCh was full (256 buffer). There was no retry — the comment "next progress event will retry" was false. Now uses blocking send with 10s timeout.
- **`liveIterationBlocks` deduplicates tools by `Name + Label`** — but `generating` tools skip dedup. Streaming tools have no labels yet (args still being generated), so two same-name tools (e.g. two `Read` calls) would share the same `Name+""` key and the second gets dropped. Fix: `Status=="generating"` tools are appended unconditionally, bypassing the `seen` map.
- **`snapshotIterationChange` data mismatch path now creates a snapshot instead of discarding.** When `prev.Iteration != m.lastSeenIteration`, instead of skipping the snapshot entirely (which permanently loses iteration data), it creates a snapshot tagged with `prev.Iteration` after checking it wasn't already snapshotted.
- **`mergeSubAgentTrees` preserves children when `new.Children` is empty but `prev.Children` exists.** Instead of returning nil (dropping the subtree), it marks all prev children as done via `markAllDone()`. This prevents child progress trees from flickering when the server doesn't include children in every update frame.

### Reasoning Contamination
- **`snapshotIterationChange` must NOT use `prev.ReasoningStreamContent`/`prev.StreamContent` as fallback.** Stream-only messages (`StreamReasoningFunc`/`StreamContentFunc`) update `m.progressState.current.ReasoningStreamContent`/`StreamContent` directly between structured progress updates. Since `prev` is captured at the START of `handleProgressMsg` (pointer to the same struct), `prev.ReasoningStreamContent` can contain the NEXT iteration's reasoning when the iteration changes. Only structured sources are used in snapshotIterationChange: `prev.Reasoning` → `reasoningByIter` → `m.lastReasoning` → `msg.payload.Reasoning`. If structured Reasoning is unavailable (progressCh coalescing), the snapshot has empty reasoning — better no reasoning than wrong reasoning.
- **Ctrl+C cancel path MUST capture streamed content (`StreamContent`/`ReasoningStreamContent`) as snapshot fallback.** When Ctrl+C interrupts mid-stream, the LLM hasn't finished — `recordAssistantMsg` hasn't set `ThinkingContent`/`ReasoningContent`, so structured fields are empty. The only data available is the streamed content in `prev` (handleProgressDone cancel path) or `m.progressState.current` (cancelledTurnIterations). Using stream fields here is SAFE and CORRECT because we're snapshotting the CURRENT iteration — prev/current IS the live progress the user was watching. This differs from snapshotIterationChange where stream fields are contaminated by the next iteration. Without this fix, the last reasoning block and content vanish after Ctrl+C (they were only in stream fields). Priority in cancel path: structured `prev.Thinking` → `prev.StreamContent` for content; `reasoningByIter` → `lastReasoning` → `prev.Reasoning` → `prev.ReasoningStreamContent` → `msg.payload.Reasoning` for reasoning.
- **Reasoning stream without iteration advance contaminates previous snapshot.** `isStreamOnly` reasoning updates bypass `snapshotIterationChange`. When the next structured progress arrives with a new iteration, the old progress (with accumulated reasoning) gets snapshotted under the OLD iteration. Fix: `advanceIterationForReasoning()` detects when reasoning arrives for a completed iteration and advances `m.progress.Iteration`. Must also call after `restoreIterationHistory` + `carryForwardProgressState` for the TUI restart case.
- **`handleAgentMessage` must NOT use `m.progress.ReasoningStreamContent` as `m.lastReasoning`/`m.reasoningByIter` fallback.** When the agent's text reply arrives after a structured progress has advanced `m.progress.Iteration` (e.g. #3→#4), `ReasoningStreamContent` may still contain the previous iteration's reasoning content. This causes misattribution: the previous iteration's reasoning gets stored under `m.reasoningByIter[newIter]`, and the next iteration's snapshot will show the wrong reasoning text (symptom: "#4 shows #3's opening content"). Only use `m.progress.Reasoning` (the authoritative per-iteration field, set at LLM completion).
- **`handleSuHistoryLoad` acceptProgress assistant merge must NOT cross turns.** When TUI restarts mid-turn, the code merges the last history assistant into `startAgentTurn()`'s empty streaming slot. The old code searched backward for `role=="assistant"` and always merged — but if there's a `role=="user"` message between the found assistant and the streaming slot, the assistant belongs to a PREVIOUS turn. Merging it displays the previous turn's content as the current streaming message (Bug: stale content appears below latest iteration on restart). Both merge branches (the `canMerge` path and the `streamingMsgIdx < 0` path) must scan for intervening user messages before merging.
- **`StreamingTools` is a stream-only field — same protection rules as `StreamContent`/`ReasoningStreamContent`.** It carries tool names detected during LLM streaming (Status="generating"), before arguments finish generating. Must NEVER enter `snapshotIterationChange` or any structured snapshot path. `isStreamOnly` must check `len(StreamingTools) > 0` and merge into `current.StreamingTools`. `CollectStreamWithCallback` now has 5 params (added `onToolCall func([]ToolCallDelta)`) — fires once per tool name arrival (first chunk), NOT per argument delta. `GenerateStreamAndCollect` and `generateResponse` signatures also gained the matching param. `buildStreamCallbacks` returns a third callback that converts deltas to `[]ToolProgress{Status:"generating"}` and broadcasts via `SendProgress` (not `SendStreamContent`, which only supports content/reasoning strings).

### CLI Rendering Performance
- **`renderProgressBlock` must use `padProgressLines` instead of `blockStyle.Render`.** `lipgloss.Style.Render` runs `ansi.StringWidth` on every character — O(total_content) per tick. Since progress block content is already hard-wrapped to `innerWidth`, manual `" " + line + " "` padding produces identical visual output at O(lines) cost. This was the #1 CPU consumer at 55% of total in profiles with many iterations.
- **`progressBlockCompositeFP` must include `m.ticker.ticks` for spinner animation.** Without ticks in the fingerprint, the cache stays stable across 100ms ticks (elapsed is quantized to whole seconds), freezing the spinner/pulse animation. The composite FP combines pre-computed sub-block FPs + ticks + active tools for O(active_tools) instead of O(total_content) full-string hashing.
- **Progress history rendering must be incremental, not full rebuild.** `renderHistoryRange` renders only newly added iterations (`cachedProgressHistory[oldLen:]`) and appends to cache. Old code re-rendered all N iterations on every new iteration — O(N) spike proportional to iteration count. The three-branch logic: cache hit (O(1)) / incremental append (O(new_iters)) / full rebuild (only on width change).
- **Tick path must use direct line array assembly, not `setViewportContent` string path.** `cachedHistoryLines` + `cachedProgressBlockLines` + `cachedDynamicLines` are directly `append`ed into `[]string` and passed to `viewportSetLinesBypassMaxWidth`. This avoids O(total_content) string concatenation + O(N_iters) `lipgloss.Width` + `wrapPreservingGuide` per tick. Scroll behavior (atBottom/GotoBottom/newContentHint) is preserved identically.
- **`syncProgressTodos` must do change detection before calling `relayoutViewport`.** High-frequency progress events carry the same Todos every time. Without `todosEqual()` check, each event triggers `relayoutViewport` → `fullRebuild` → `glamour.Render` + `chroma.Highlight` on ALL messages — ~34% CPU waste during agent work. Only call `relayoutViewport` when todo count changes; for content-only changes (e.g. item marked done), just set `renderCacheValid = false`.
- **`relayoutViewport` only invalidates render caches when width changes.** Height-only changes (todo bar, panel open/close, `endAgentTurn`) just resize the viewport without rebuilding all messages. Without this guard, every `endAgentTurn` / `handleProgressDone` triggers O(N) `fullRebuild` with `glamour.Render` on every message — accounts for ~36% CPU during spike periods.
- **`viewport.SetContent` calls `maxLineWidth` internally which runs `ansi.StringWidth` on every line — ~49% CPU with large content.** `cli_viewport_bypass.go` bypasses this by directly setting viewport's private `lines` and `longestLineWidth` fields via `unsafe` (offsets computed via `reflect.FieldByName` at init). **The caller must pass the REAL measured max width, NOT cw.** Viewport's `visibleLines()` trusts `longestLineWidth <= maxWidth` to skip truncation. If `maxW = cw` but some wrapped lines are actually wider (hardwrap edge cases, glamour output), those lines escape truncation, then lipgloss v2's `Wrap()` in `viewport.View()` re-wraps them, producing extra rendered lines that overflow the viewport height and push the input box off screen. `setViewportContent` tracks actual `maxW` via `lipgloss.Width()` during the wrap loop; `cachedHistoryMaxWidth` caches this for the fast path.

### CLI Race Conditions (Remote Mode Session Switching)
- **`handleSuHistoryLoad` must NOT clear `suLoading` before the stale check.** The old code set `m.suLoading = false` unconditionally on line 876, BEFORE checking `msg.chatID != m.chatID`. A stale RPC callback from the old session clears the new session's loading guard, letting progress events through prematurely and corrupting typing/idle state. The `suLoading=false` now only executes after the stale guard passes.
- **`handleHistoryReload` needs `channelName`/`chatID` stale check.** `cliHistoryReloadMsg` had no session identifiers — an old session's async history load could overwrite `m.messages` after the user already switched away. Added `channelName`/`chatID` fields to the message and a guard in the handler.
- **`splashTickMsg` must carry `gen` (tickGen) for stale rejection.** Rapid session switching creates multiple splash tick chains. Without gen checking, old ticks from cancelled sessions compete with new ticks, causing animation jitter and potential state corruption. Handler now returns early when `msg.gen != m.tickGen`.
- **`handleSuHistoryLoad` default branch does NOT restore `lastTokenUsage` — context bar shows 0 on first switch to idle session.** The `acceptProgress` branch restores token usage from `activeProgress.TokenUsage`, but when the target session has no active turn, the `default` branch runs and leaves `lastTokenUsage` nil. Fix: `suLoadHistoryCmd` now fetches token state via `GetTokenStateFn` and `suHistoryLoadMsg` carries `tokenPrompt`/`tokenCompletion`; the handler falls back to this when `lastTokenUsage` is still nil after processing.
- **Remote reconnect handler must NOT SetProcessing(true) when progress.Phase == "done".** `GetActiveProgress` may return a stale snapshot whose Phase is already "done" (server hasn't cleaned it up yet). Calling `SetProcessing(true)` on a done turn causes the TUI to start a new empty turn (spinner + streaming slot) with no further progress events arriving — permanent spinner. Fix: only `SetProcessing(true)` when `progress != nil && progress.Phase != "done"`.

### TUI Control & Config Tools (AI-Native)
- **`SyncLayoutSettings` and 5s polling goroutine REMOVED.** Layout sync is now event-driven: `ApplyInitialLayout` at startup, `doSaveSettings` → `cliSettingsSavedMsg` for user changes, `tui_control` for AI changes. `refreshRemoteValuesCache` runs ONCE at startup, not every 5s. `valuesCache` is updated on-demand by settings panel save, subscription switch, and config tool set callbacks. Adding polling back will cause ~30% CPU waste from unnecessary `invalidateAllCache` → `fullRebuild` cycles (7+ GB/min temporary string allocations observed).
- **`relayoutViewport` → `newGlamourRenderer(cw - 4)` must only be called from the event loop goroutine.** The glamour renderer is NOT goroutine-safe. Background goroutine calls cause concurrent access to `m.renderer` and crash.
- **`sidebar_width` and other layout keys are NOT in `config.Config` struct.** `SaveToFile` deep merge only PRESERVES unknown keys from disk, never writes new ones. Use `saveLayoutToConfig()` (raw JSON read-modify-write) to persist these keys to `config.json`.
- **Remote `tui_control` flow**: Server → `tui_control_req` WS message → client `readPump` → `tuiControlReqCb` → `SendTUIControl` → `asyncCh` → `handleAsyncDrain` → `program.Send` → event loop → `handleSessionControlMsg`. The WS callback in `transport_remote.go` MUST be wrapped in `go func()` to keep `readPump` responsive — otherwise RPC calls within handlers (e.g. `SessionsDeleteFn`) deadlock waiting for responses.
- **`config` tool uses `SettingsSvc` auto-injection, not Agent callbacks.** `buildToolContext` injects `ConfigGet`/`ConfigSet` from `cfg.SettingsSvc`. Runner CRUD callbacks (`RunnerCreate`/`RunnerList`/`RunnerDelete`/`RunnerGetActive`/`RunnerSetActive`/`RunnerRename`) are injected from `tools.RunnerTokenStore` via `tools.GetRunnerTokenDB()`. Works in ALL modes (local + remote via RPC). Do NOT rely on Agent `SetTUICallbacks` for config.
- **Runner switch is session-level, not user-level.** `config switch` writes to `SandboxRouter.sessionRunners` (in-memory sync.Map keyed by `"channel:chatID"`). Per-tool-call re-resolution in both `buildToolExecutor` (engine_wire.go) and `defaultToolExecutor` (engine.go) calls `SandboxForSession` before each Execute. All tools (Shell, Read, Grep, Glob, FileReplace) immediately route to the new runner. CWD auto-resets to runner's live workspace.
- **`config` tool masks sensitive keys on read** (`api_key` → `sk-a***`, `runner_token` → first 4 chars + `***`). Writes are NOT blocked — users can type API keys anyway, our responsibility ends at masking.
- **`DeleteTenant`/other `TenantService` methods must nil-guard `s.db`.** Interactive session cleanup races with DB initialization — `destroyInteractiveSession` can fire before the DB is connected, causing nil pointer dereference.

### Hooks System
- **Old `ToolHook`/`HookChain` is gone.** Replaced by `agent/hooks/Manager`. Any code referencing `HookChain`, `ToolHook`, `executeWithHooks` is stale.
- **Manager.Emit() is shared across Agent + SubAgents** (same instance). Must be concurrency-safe.
- **Decision priority**: `deny > defer > ask > allow`. Low-priority layer deny cannot be overridden by high-priority allow.
- **Script plugin triggers are global hooks** (`hookRegistration.Global=true`). `bridge.Dispatch` skips session isolation for them — they manage per-workDir state and must fire for all sessions. Without this, multi-session remote CLI silently drops triggers for all but the last session that called `RefreshWorkDir`.

### Backend/Transport Architecture
- **AgentChannel processing loop MUST be concurrent (each request in its own goroutine).** The old serial processing loop (`case req := <-ac.inbox: runFn(...)`) caused circular-wait deadlocks when two agents SendMessage to each other — A's runFn blocks waiting for B's reply, but B's inbox can't be processed because A's loop is stuck in runFn. Fix: dispatch each request to `ac.wg.Go(func() { processRequest(req) })`.
- **AgentChannel.Send() MUST respect caller context via `msg.Ctx`.** `OutboundMsg.Ctx` (json:"-") carries the caller's context so `Send()` can detect Ctrl+C/tool-timeout. Without this, `Send()` only checks `ac.ctx.Done()` (from `context.Background()`), making Ctrl+C unable to unblock pending RPCs. `Dispatcher.SendMessageCtx(ctx, ...)` sets `OutboundMsg.Ctx`. `MessageSenderCtx` interface extends `MessageSender` with context-aware sending. `sendMessageWithCtx()` in send_message.go uses type assertion to prefer `MessageSenderCtx`.
- **AgentChannel.StartWithContext(parentCtx) binds lifecycle to parent.** When the parent agent is cancelled (Ctrl+C → `agentCtx` cancelled → `Agent.Close()`), all AgentChannels stop. Use this for agent-owned channels instead of plain `Start()`.
- **`registerAgentChannel` callback creates AgentChannel with `Start()`.** For parent-context binding, use `StartWithContext(agent.agentCtx)` — but note `agentCtx` is only set during `Run()`. Current code uses `Start()` and relies on `Dispatcher.Stop()` during shutdown to clean up. The caller-context cancellation via `msg.Ctx` is the primary Ctrl+C propagation mechanism.
- **CLI always uses Client → Transport → Server, even in local mode.** Client (agent/client.go) is a pure RPC client — every method is `Transport.Call("method", params)`.
- **Transport is the pure transmission layer** (agent/transport.go). Only 2 methods: `Call(method, payload)` and `Close()`. `ChannelTransport` (agent/transport_channel.go) dispatches directly to `ServerCore.HandleRPC`. `RemoteTransport` (agent/transport_remote.go) sends JSON-RPC over WebSocket.
- **`connect()` replaces `t.conn` with a new WebSocket — old `readPump` must NOT clear the new connection.** Old `readPump` hits `ReadMessage` error after `old.Close()`, then unconditionally sets `t.conn = nil` and fires a second reconnect — causing an infinite reconnect loop, dropped `BindChat`, and permanent spinner. Fix: `readPump` checks `t.conn == conn` before clearing; if `connect()` already replaced the connection, old `readPump` returns silently without clearing or signalling.
- **`setConnState` must NEVER be called while holding `t.connMu`.** `setConnState` internally acquires `t.connMu.Lock()`. Go `sync.Mutex` is NOT reentrant — calling it inside the same-goroutine `connMu` critical section deadlocks the entire event loop (TUI frozen, Ctrl+Z broken). Fix: set a local flag inside the lock, call `setConnState` after `Unlock()`.
- **`RemoteTransport` events (conn_state, reconnect) must bridge to `Client`'s subscriber base.** `RemoteTransport` and `Client` each create their own `baseTransport` instance. Events emitted by `setConnState`/`reconnectLoop` on `RemoteTransport.baseTransport` never reach subscribers registered via `app.client.Subscribe()` (which uses `Client.base`). Fix: `Client.Start()` calls `ShareEventBase(&c.base)` on the transport; `RemoteTransport.emitLocal()` dispatches on the shared base. Any new local event in RemoteTransport must use `emitLocal`, not `t.emit()`.
- **Disconnect detection must cover write failures, not just read errors. And the TUI must react immediately.** Fix: (1) transport: `sendPing`/`SendMessage` clear `t.conn` on write failure. (2) TUI: `sendInbound()` immediately sets `m.connState = "disconnected"` when write fails. **CRITICAL: The `sendInboundFn` callback in main.go was fire-and-forget (`clipanic.Go` goroutine) and ALWAYS returned `true` — making ALL TUI-layer checks dead code. Every `sendInbound`/`sendMessage`/`sendToAgent` fix was unreachable because the callback never reported failures. Fix: `sendInboundFn` must call `SendInbound` synchronously and return the real result. Also: `sendMessage()` (main path from handleEnterKey) AND `sendToAgent()` (slash commands) both check return value before `startAgentTurn()`. (3) View() reconnect overlay must come BEFORE `suLoading` — stale `suLoading` from a dead server permanently blocks the splash.**
- **FINAL ARCHITECTURE: `showDisconnect` removed — `connState` is the single source of truth.** All splash, disconnect guard, and tick conditions use only `m.connState`. `SetConnState` writes `connState` directly to model (bypasses all channels). `sendInboundFn` is async (`clipanic.Go`) and does NOT touch `connState` — transport layer manages state independently. `handleSuHistoryLoad` skips redundant RPC when history already provided in message. **`readPump` must start BEFORE `emitLocal(ReconnectEvent{})`** — otherwise `BindChat`/`GetHistory` RPCs in the handler have no reader and timeout after 30s, freezing the TUI. No deadlocks, no race conditions, no redundant flags.
- **ServerCore** (serverapp/server_core.go) is the shared server core for both local and remote modes. Owns Agent + RPCTable + Dispatcher + Bus. Created by CLI (local mode) and server.go (remote mode).
- **RPCTable is the single handler truth source** (serverapp/rpc_table.go). Handlers access `h.Ag` (the *Agent) directly — no intermediate Backend/DirectBackend layer.
- **Lifecycle interfaces** (agent/lifecycle.go): AgentRunner, EventRouter, CallbackRegistry — only used by the legacy Backend struct for remote mode.
- **Adding a new RPC method**: 1 handler in rpc_table.go (using h.Ag) + 1 method in client.go (Transport.Call) + 1 MethodXxx constant in req_types.go.
- **Adding new transports** (gRPC, MCP) only requires implementing 2 Transport methods (Call + Close).
- **ChannelPluginTransport (`agent/transport_channel_plugin.go`) is a separate concept from Transport.** It wraps a channel plugin process's stdin/stdout as a bidirectional JSON-RPC channel (not an agent Transport). The plugin acts as a full RPC client (like remote CLI over WS). Created by `stdioChannelPluginProvider` (`serverapp/channel_plugin.go`) which spawns a **dedicated** process — NOT the same process used for plugin activation.
- **`ChannelProviderFactory` replaces the old `GrpcChannelBridgeFactory`.** Defined in `plugin/channel_provider.go`. The factory is set by serverapp during init with a lazy RPCTable reference (plugins activate during `InitServer` before RPCTable exists).

### Channel Configuration
- **RPC responses MUST NOT be dropped.** `readPump` processes RPC synchronously in `web.go`. The old code used `select { case sendCh <- msg: default: drop }` for RPC responses. During `plugin_reload_all`, multiple widget pushes (`PushPluginWidgetsPerSession`) can fill `sendCh` (64 buffer) while the RPC handler runs. When the handler completes, its response hits a full `sendCh` and gets silently dropped → client blocks until 30s timeout. Fix: block with `time.After(10s)` instead of `default` drop. Also `writePump` must set `SetWriteDeadline(30s)` on all `WriteJSON` calls — without it, network back pressure causes `WriteJSON` to block indefinitely, freezing `writePump` so `sendCh` never drains.
- **Channel-scoped tools via `Registry.channelTools`.** Feishu tools (Card + Feishu MCP) are registered with `RegisterForChannel("feishu", tool)` instead of global `Register(tool)` + `SupportedChannels()` filtering. This eliminates the O(N) per-prompt `visibleToolDefs` overhead and keeps `globalTools` clean. `AsDefinitionsForSession` merges `channelTools[ChannelFromSessionKey(sessionKey)]` automatically. Tool execution uses `GetForSession(name, tenantID, sessionKey)` which checks channel → tenant → global.
- **Channel plugin tools via `channel_tools` protocol.** Channel plugin process sends `{"type":"channel_tools","tools":[...]}` to declare tools. `ChannelPluginTransport.handleChannelTools()` wraps each in `ChannelToolBridge` and registers via `RegisterForChannel`. Tool execution proxies via `Call("execute_tool")` RPC to the channel process. Hot-update: new `channel_tools` message replaces the entire set.
- **Channel plugin prompt via `channel_prompt` protocol.** Channel plugin process sends `{"type":"channel_prompt","system_parts":{"05_channel_xxx":"..."}}` to declare channel-specific system prompt. `ChannelPluginTransport.handleChannelPrompt()` stores parts in `channelPluginPromptProvider` (thread-safe via `sync.RWMutex`), then fires `OnChannelPrompt` callback → `Agent.AddChannelPromptProvider()` registers with `ChannelPromptMiddleware` (priority 5). `stdioChannelPluginProvider` wires `agentGetter` lazily — the `OnChannelPrompt` callback resolves the Agent at runtime. Hot-update: new `channel_prompt` replaces the entire parts map. Key naming convention: `"05_channel_xxx"` prefix (after `"00_base"`, before `"10_skills"`).
- **`RegisterForChannel` empty channel falls back to global.** Passing `""` as channel calls `Register(tool)` instead.
- **`ChannelFromSessionKey` is public.** Used by `AsDefinitionsForSession` and `GetForSession` to extract channel from `"channel:chatID"` format.
- **TUI channel config changes require live channel restart.** Writing config.json is not enough — Feishu/Web/QQ/NapCat channels are created once at startup via `registerChannels()`. `SetChannelConfig()` now calls `reconfigureFn` (set by server.go) to start/stop the affected channel. Any new channel type must be added to both `channelShouldRun()` and `createChannelInstance()`.
- **Key naming must be consistent across all channels.** Web used `enable` while Feishu/QQ/NapCat used `enabled` — the RPC handler only checked `enabled`, so web changes were silently ignored. All channels now use `enabled` (with backward compat for `enable`).
- **New client methods must be added to `*Client` in `agent/client.go`.** The `Client` struct is the unified RPC client used by both CLI and server modes. Both local (ChannelTransport) and remote (RemoteTransport) modes go through the same RPC path.
- **Command hooks disabled by default** — requires `enable_command_hooks: true` in config.
- **Max 10 handlers per event**, total timeout 60s. Excess silently truncated with warning log.
- **Feishu interactive card has a ~50 element limit (error 11310 "element exceeds the limit").** Every `form`, `select_static`, `button`, `input`, `markdown`, `hr`, `column_set` counts as an element — nested form children count too. The model settings tab hit this limit with 3 independent tier forms (each = 1 markdown + 1 form + 1 select + 1 button = 4 elements × 3 = 12 + separators). Fix: consolidated 3 tier forms into 1 form with 3 selects + 1 submit (5 elements total). Always validate card JSON via Feishu send_message API before deploying — the error is `230099 / ErrCode: 11310`, not the usual `200673`.
- **Feishu settings card `settings_select_submit` uses select name == form name convention.** Every `select_static` inside a form shares its name with the enclosing form, so the handler reads the selected value directly from `formStr(actionData, formName)`. No hardcoded select-name list. The tier form (`tier_form`) is a special case: it has 3 selects (`tier_vanguard_select` / `tier_balance_select` / `tier_swift_select`) routed to `settings_set_all_tiers` which reads all 3 from form_value on a single submit.
- **`LLMCallbacks` max_context/max_output signatures take explicit `(subID, model)`.** `LLMGetMaxContext(senderID, subID, model)` / `LLMSetMaxContext(senderID, subID, model, maxContext)` — same for MaxOutputTokens. When subID/model are non-empty (feishu model tab), the callback writes per-(subID, model) PerModelConfig directly via `Agent.*ForSubModel` methods. When empty (web legacy), falls back to session resolution. Channel UIs MUST pass the (subID, model) from the model selector — never resolve subscription from model name alone (project policy). `settings_set_model` handler rejects option values without the `subID|model` separator.

### Plugin System
- **Plugin system is opt-in** — only activates when `plugins.enabled: true` in config.json. No plugin loading happens without explicit user consent.
- **`pm.workDir` is `atomic.Value` (not `string`).** `activate()` may be called while `pm.mu` write lock is held — reading workDir must be lock-free. Never change it back to `string` or `activate`/`InstallPlugin` will deadlock.
- **`runAndUpdate()` does NOT write global slot cache.** It calls `NotifyUpdated()` instead of `UpdateWidget()`. Writing global cache causes cross-session overwrites (session B's git branch overwrites session A's).
- **CLI WS clients must NOT auto-subscribe to senderID ("admin").** `client_type=cli` connections skip p2p subscribe. Subscribing CLI to "admin" causes `PushPluginWidgetsPerSession` to send stale content to wrong windows.
- **`PushPluginWidgetsPerSession` skips non-path chatIDs.** Only chatIDs starting with `/` are session chatIDs. Virtual chatIDs like "admin" or "web-123" are not rendered.
- **`OnPluginWidgets` callback filters by chatID.** Client-side rejects pushes for other sessions. Double protection against cross-session widget corruption.
- **Script plugin outputs map is per-workDir.** `RenderForWorkDir(width, workDir)` reads `outputs[workDir]`. `Render()` falls back to shared `pctx.WorkingDir()` — never use for remote multi-session.
- **`HookPayload.ToolOutput` is truncated to 8KB.** Don't rely on it for full file content. Plugins needing full output should use dedicated tool result channels.
- **PluginManager.ActivateAll() collects capabilities; WireAll() connects them.** Never call registerCapabilities manually — WireAll is the single integration point.
- **PluginEntry.stateMu protects state transitions.** Use CAS pattern (check state → set activating → set active/error) to prevent concurrent activation races.
- **gRPC plugin processes are killed on timeout/cancellation.** The `call()` method kills the process and marks it as not-running to prevent goroutine leaks from blocked stdout reads.
- **PluginToolBridge auto-detects PluginToolV2.** If a plugin tool implements V2, the bridge passes ToolCallContext. Otherwise falls back to V1 Execute(ctx, input).
- **Plugin IDs validated with regex `^[a-zA-Z0-9][a-zA-Z0-9._-]{0,127}$`.** This prevents path traversal, null bytes, and injection attacks in storage paths.
- **Storage files use 0600 permissions and atomic write (tmp+rename).** Never use 0644 for plugin storage.
- **WASM runtime is skeleton-only.** It compiles and loads but Activate() is a no-op. Phase 2 requires wazero dependency.
- **PluginContext provides 4 type-safe Storage helpers:** `StorageInt`, `StorageBool`, `StorageJSON`, `StorageGetJSON`. These wrap the base `StorageAccessor` with parse/unmarshal and return typed results. Failed parses return zero-value + false (not errors) for Int/Bool, and errors for GetJSON.
- **Auto-retry runs in a background goroutine.** `SetAutoRetry(true, maxRetries)` starts `retryLoop` with exponential backoff (1s→30s cap). **`DeactivateAll()` cancels the retry context** — if you call `activate()` manually after `DeactivateAll()`, you must re-enable auto-retry or failed plugins won't recover automatically.
- **Manifest `timeout` field accepts Go duration strings** (`"30s"`, `"1m"`, `"500ms"`), parsed via `time.ParseDuration`. Empty or missing defaults to `DefaultPluginTimeout` (30s). Max allowed: 5 minutes.
- **EventBus requires `bus.plugin` permission** in addition to `bus.read`/`bus.write`. Subscribe needs `bus.plugin` + `bus.read`; Publish needs `bus.plugin` + `bus.write`. This separates plugin-to-plugin events from the core message bus.
- **`InstallPlugin` uses `filepath.EvalSymlinks`** to resolve the real directory path before deletion check, preventing symlink-based path traversal attacks. Only directories under `xbotHome` are deleted.
- **`WatchConfig` polls config.json every 30 seconds** (configurable, min 5s). It compares `plugins.disabled_plugins` lists via diff and reactively deactivates newly disabled / activates newly enabled plugins. Returns a stop channel for shutdown.
- **DependencyResolver uses Kahn's algorithm (BFS topological sort).** Circular dependencies return an error (not panic). `AddManifest` with duplicate ID replaces the existing entry. Resolve() returns activation order — plugins with no dependencies first, then in dependency order.
- **Profiler is safe for concurrent use** (sync.Mutex). `Profile(pluginID)` returns a **copy** of PluginProfile — safe to mutate without affecting internal state. Unprofiled plugins return zero-value PluginProfile.
- **ExportConfig acquires RLock on PluginManager.** Must not be called while holding a write lock (e.g., inside custom Activate/Deactivate that calls pm.mu.Lock). ImportConfig acquires write lock internally — do not nest inside another write-locked operation.
- **MockPlugin/MockTool chain API returns the same pointer** — each `With*` call mutates and returns `*MockPlugin`/`*MockTool`. Do not share a single mock across parallel tests without cloning.
- **PluginRegistry MVP only supports local sources** for installation. Search operates on locally installed plugins only. GitHub/URL sources are defined but InstallFromSource is not yet implemented — Phase 3 scope.
- **Plugin migration `Migrator` creates backup before applying migrations.** Backup is stored in `~/.xbot/plugins/<id>/backups/<version>/`. Rollback restores from the most recent backup. Migrations run sequentially by version order.
- **`pluginLogger` writes ONLY to per-plugin log file — NOT to global logrus.** Plugin operational logs (tool/hook registration, script execution, widget refresh) are isolated in `~/.xbot/plugins/<id>/logs/`. This keeps the main xbot log clean. Only framework-level lifecycle events (discovered/activated/deactivated/failed) use global `log.Xxx()` in `manager.go`. Script runtime logs go through `p.logger()` (helper with noopLogger fallback for tests). If `fileOut == nil` (writer creation failed), `pluginLogger.emit()` falls back to global logrus so no logs are lost.
- **`toolHint` zone plugins run synchronously on PostToolUse hook.** When `isHintPlugin=true`, the hook trigger runs `runScript` inline (not via triggerCh). The engine calls `PluginManager.GetToolHints()` immediately after the hook returns to populate `ToolProgress.ToolHints`. **`GetToolHints()` consumes (clears) the hint after reading** to prevent stale content from attaching to the next tool.
- **`snapshotIterationChange` must include ActiveTools(done).** When an iteration ends, completed tools may still be in `ActiveTools` (status=done) rather than `CompletedTools` (which is populated later by `progressFinalizer`). Only checking `CompletedTools` loses ToolHints data.
- **Do NOT use glamour to render diff inside progress panel.** Glamour's output (background fills, margins, line wrapping) corrupts the progress panel border layout. Use direct ANSI coloring (`renderDiffANSI`) with width truncation instead.
- **`runScript` must `os.Stat(workDir)` before setting `cmd.Dir`.** On Windows parallel tests, temp dirs may be cleaned up before the script runs, causing `chdir` failure. If dir doesn't exist, skip setting `cmd.Dir` and run in plugin's own directory.
- **`runScript` injects `XBOT_WIDGET_ID` env var.** When a plugin declares multiple widgets, each widget gets a `widgetAdapter` that passes its `widgetID` to `runScript`. Scripts can branch on `$XBOT_WIDGET_ID` to produce different output per widget. Without this, all widgets of the same plugin produce identical output (the reported "5 duplicates of `main +10 ↑13`" bug). The `outputs` cache is a two-level map: `workDir → widgetID → output`.
- **Platform-specific entry points.** `PluginManifest` supports `entry_windows`, `entry_darwin`, `entry_linux` fields that override the generic `entry` for matching OS. `scriptPlugin.resolvedEntry()` selects the correct entry at runtime. Use this when a plugin needs bash on Unix but PowerShell on Windows. Generic `entry` is always the fallback.
- **Session context env vars for plugins.** `runScript` injects `XBOT_MODEL`, `XBOT_MAX_CONTEXT`, `XBOT_TOKEN_USAGE` (format: `"prompt/completion"`), `XBOT_PROMPT_TOKENS`, `XBOT_COMP_TOKENS`. Data flows via `hooks.SessionContext` (mutable pointer in context, updated by engine after each LLM call and compression) → `plugin_bridge.go` extracts into `HookPayload.Extra` → `runScript` serializes to env vars. Widgets can display model name, context usage bar, and token costs.

### Windows
- `syscall.PROCESS_QUERY_LIMITED_INFORMATION` and `STILL_ACTIVE` not in Go stdlib — define as uint32 constants.
- `exec.ExitError.ExitCode()` is cross-platform; avoid `syscall.WaitStatus` type assertion.
- `signal.Notify(sigCh, syscall.SIGTSTP)` doesn't compile on Windows — use build-tagged files.
- PowerShell env output is newline-delimited, not null-delimited.

### Worktree
- **`RegisterPeer` always uses role="peer" — no primary concept.** Every session is an equal peer for awareness purposes. `WorktreeTool.init` and `AutoDetectAndInit` both create physical worktrees for each session. Only worktree creation requires `git worktree add` which uses `--detach HEAD` (works on dirty trees too).
- **`createWorktree` uses `--detach` not `-b`.** `git worktree add --detach HEAD` followed by `checkout -b` in the worktree. This avoids dirty-tree failures that `-b` would trigger.
- **AutoDetectAndInit runs in `buildPrompt` (not `processMessage`).** `buildPrompt` is the common code path for ALL message types. Session CWD is updated before system prompt construction.
- **`IsWorktreeIsolated` overrides `isUnrestricted()`.** CLI mode (`sandbox="none"`) normally bypasses all path checks, but when `IsWorktreeIsolated=true`, `isUnrestricted()` returns `false`.
- **Worktree paths must be outside main repo.** Git rejects `git worktree add` inside the main working tree. Paths: `{repo}/../.xbot-worktrees/{role}-{instance}/`.
- **Worktree creation is serialized via `GlobalWorktreeRegistry.mu`.** Two agents creating worktrees simultaneously would race on `.git/worktrees/` lockfiles.
- **Registry persisted per-repo to `{repo}/../.xbot-worktrees/registry.json`.** `saveRepoLocked` uses atomic tmp+rename. `ensureLoaded` lazily restores on first access. Orphaned worktree directories are skipped on load.
- **`BuildSystemReminder` takes `sessionKey` parameter.** Used to query `GlobalWorktreeRegistry` for worktree/peer info injected into sys_reminder per iteration.
- **`MethodSetCWD` only sets CWD when empty.** Prevents CLI session switch from overwriting worktree CWD with main workspace path. Session-specific CWDs (worktree, Cd tool) are preserved.
- **`go:embed embed_skills/*` auto-discovers new skills.** Adding a directory under `tools/embed_skills/` requires zero code changes.
- **`buildPrompt` 绝不恢复 CWD 到 worktree 路径。** 当 session 已注册后 `AutoDetectAndInit` 跳过，这是正确行为。agent 通过 Cd 工具主动切换的目录必须被尊重，如果在后续 buildPrompt 中强制恢复会破坏 agent 的 Cd 操作，产生人机对抗。
- **Cd 工具绝不强制 worktree 边界。** CLI 模式下 Cd 允许 agent 自由导航到任何目录，包括主仓库。如果对 Cd 加边界限制会导致 agent 无法读取 worktree 外的文件（如 AGENTS.md、知识库、配置文件），这是严重 bug。worktree 隔离是软性的工作指引，不是硬性监狱。
- **新会话绝不能进入老 worktree（纵深防御 6 层）。** `loadPersistedCWD` 拒绝加载 worktree 路径和不存在目录 → `autoDetectAndInitInto` 拒绝从 worktree CWD 创建 worktree + 自动清理 worktreeDir 不存在的孤儿条目 → `buildPrompt` 验证 worktree 所有权（不匹配则重置到 workspaceRoot）→ `showSessionCreateDialog` 和 `/chat new` 新建前清理 worktree registry + persisted CWD + savedSessions + todos。详见 `docs/agent/worktree.md`。

## Development Principles

### Never Blame the User's Binary

**永远不假设用户用了旧二进制。** 如果怀疑版本问题，说明自己的排查逻辑有漏洞，不是用户的问题。

### Always Prefer Explicit

**核心原则：永远优先使用显式 API，避免隐式假设。**

本项目遵循 "always prefer explicit" 开发原则。大量 bug 源于隐式设计——调用者无法从 API 签名推断出所有必要参数或行为。

#### 具体实践

1. **避免直接使用结构体作为公共 API 参数**
   - ❌ `func NewFoo(cfg FooConfig) *Foo` — 调用者可能遗漏 `FooConfig` 中的关键字段
   - ✅ `func NewFoo(opts ...FooOption) *Foo` — 使用私有结构体 + 构造函数 + 显式 Option 模式
   - ✅ `func NewFoo(required string, optional ...string) *Foo` — 必填参数显式列出

2. **假设调用者只看到你的 API 签名**
   - 调用者没有义务阅读实现细节
   - API 签名应自解释：参数名、类型、顺序应清晰表达意图
   - 使用 `// WithXxx` 风格的 Option 函数提供可选配置

3. **宁可冗长，不要隐晦**
   - 5 个显式参数优于 1 个包含 20 个字段的结构体
   - 如果必须用结构体，确保必填字段在构造函数中强制提供
   - 使用 `Must` 前缀函数（如 `MustParse`）在编译期捕获错误

4. **文档即合同**
   - 每个公共函数/类型必须有 godoc 注释
   - 注释应说明 "什么" 和 "为什么"，而不仅仅是 "如何"
   - 参数约束（如 "must not be empty"）应在注释中明确说明

#### 为什么重要

- 减少运行时 panic 和零值 bug
- 提高代码可读性和可维护性
- 让新贡献者能快速理解 API 用法
- 编译器帮你捕获更多错误
