# xbot

> Go AI Agent framework with message bus + plugin architecture. Supports Feishu/QQ/CLI/Web channels, tool calling, pluggable memory, skills, subagents, MCP integration.

- Build: `go build ./...` | Test: `go test ./...` | Lint: `golangci-lint run ./...`
- Config: `~/.xbot/config.json`, env var overrides
- Subscriptions: `~/.xbot/config.json` (CLI) or DB `user_llm_subscriptions` (Server) ‚Äî the single source of truth for LLM config
- Pre-commit: gofmt ‚Üí golangci-lint ‚Üí go build ‚Üí go test
- Issue templates: `.github/ISSUE_TEMPLATE/` ‚Äî YAML forms (`*.yml`) for web UI, Markdown templates (`*.md`) for CLI/AI use with `gh issue create --template`. AI agents MUST read and fill the `.md` templates (not `.yml`) since YAML Issue Forms are web-UI-only and cannot be submitted via `gh issue create --body`.

## Gotchas ‚Äî MUST READ Before Any Code Change

### Concurrency
- **Never `defer` semaphore release inside a loop.** Deadlock when iterations exceed capacity. Release immediately after Generate completes.
- Non-blocking channel sends: always use `select` with `ctx.Done()` to prevent blocking on full channels during shutdown.
- **User-scoped semaphores must not be hardcoded to capacity 1 when one sender can own multiple independent chats/sessions (for example remote CLI windows authenticated as `admin`).** Size them from configured concurrency or key them by session, otherwise different windows will block each other and look like a leaked semaphore.
- **`SetMaxConcurrency` must clear `userSemaphores` cache.** The global semaphore is rebuilt with the new capacity, but `getUserSemaphore` caches per-user channels in a `sync.Map` via `LoadOrStore`. Without `Clear()`, users with custom LLM keep using the cached semaphore with the OLD capacity forever. Symptom: setting max_concurrency to 100 has no visible effect.
- **`cancelChildSessions` must only cancel sessions with matching `parentKey`.** The old code called `cancelCurrent()` on ALL interactive sessions inside the `Range` loop before checking `parentKey`. This killed sibling/peer background agents when any single agent was unloaded or panicked ‚Äî all N peer agents die simultaneously at whatever time the first one finishes. The `parentKey` check must happen BEFORE `cancelCurrent()`.
- **SubAgents must never outlive their direct creator ‚Äî ALL completion paths need `cancelChildSessions`.** Foreground SubAgents wrap their own `fgRunCtx` (with `bgSessionCtxKey`+`bgParentKey`) so children derive context from the creator, not a grandparent. Send path `runCtx` also has these markers. Every natural completion path (foreground, bg, send) calls `runCancel()` + `cancelChildSessions(key)`. Without the context wrapper, a foreground B creating bg C would have C's context chain skip B entirely ‚Üí B finishes, C keeps running.

### Context Management & Token Estimation
- **`maybeCompress` uses ONLY API-returned `prompt_tokens` via `TokenTracker.GetPromptTokens()`.** No local estimation (tiktoken/CountMessagesTokens) in the compression decision path. The `no_data` source (no API data yet) skips all compress/masking checks. Tests must set `cfg.LastPromptTokens` to simulate a previous Run. **`maybeCompress` does NOT add `completionTokens` to its check ‚Äî `shouldCompact(totalTokens, promptBudget, threshold)` compares `prompt_tokens` against `(maxContextTokens - maxOutputTokens) * threshold`, pure prompt tokens only.** Both the context bar and maybeCompress use the same `tokenTracker.GetPromptTokens()` source ‚Äî they never diverge.
- **`ResetAfterCompress()` takes no arguments ‚Äî it zeros ALL tracker fields** (promptTokens, completionTokens, hadLLMCall). Any non-zero value causes `maybeCompress` to re-trigger immediately, creating infinite compression loops. The tracker returns "no_data" until the next real LLM API call.
- **`RecordLLMCall(prompt, completion)` takes 2 args only.** No msgCount ‚Äî the old delta estimation via `CountMessagesTokens` has been removed. Offload handles large tool results; a single iteration cannot add enough tokens to justify local estimation.
- **`RecordLLMCall` must NOT be called when `response.Usage.PromptTokens == 0`.** When a stream is cancelled mid-flight (Ctrl+C), `CollectStreamWithCallback` returns a non-nil response with zero Usage (usage events arrive at stream end). Calling `RecordLLMCall(0, 0)` overwrites the tracker's correct value from a previous iteration. This causes `SaveState` to skip (`promptTokens <= 0`), but `SaveTokenState(0, 0)` and `SaveContextTokens(0)` in `callLLM` still execute unconditionally, overwriting DB with 0. Next turn restores 0 ‚Üí context bar empty ‚Üí first callLLM corrects to real value ‚Üí immediate compression. Fix: guard all three save paths (`RecordLLMCall`, `SaveContextTokens`, `SaveTokenState`) with `response.Usage.PromptTokens > 0`.
- **`buildToolContextExtras` uses `TenantSession.MemoryService()` for `MemorySvc`/`TenantID`, NOT `LettaMemory` type assertion.** These are tenant-level fields that work for all memory providers. Only LettaMemory-specific fields (CoreMemory, ArchivalMemory, ToolIndexer) stay inside the type assertion.
- **`ObservationMaskStore` and `OffloadStore` both persist to disk.** Mask uses `~/.xbot/mask/{tenantID}/{id}.json`, Offload uses `~/.xbot/offload_store/{session}/{id}.json`. `Recall` falls back to disk on memory miss. Both cleaned on compress and `/clear`.
- **`PersistenceBridge` manages the persistence watermark (`lastPersistedCount`), not inline fields.** All compression paths use `ApplyCompress` pipeline which calls `PersistenceBridge.RewriteAfterCompress()` to atomically clear+rewrite+update watermark. The invariant `LastPersistedCount <= len(messages)` is verified by `ValidateInvariants()` at debug level after every LLM call, compression, and persistence operation.
- **Shell binary output silently bypasses offload: `summarizeShell` splits by `\n` and keeps last 5 lines.** Binary data (e.g. `cat libmujoco.so`) has very few newlines, so "last 5 lines" ‚âà entire content. The offload summary then contains nearly all original binary data, causing context explosion (39k‚Üí2.2M tokens observed). `summarizeShell` now applies `maxLineRunes=500` truncation (same as `summarizeRead`) on output lines. But any new summary generator must handle the few-lines-but-megabytes-per-line case.
- **`SanitizeMessages()` is the final validation gate before sending to LLM.** Five passes: (1) strip assistant messages with empty content AND no tool_calls ‚Äî rejected by OpenAI/Anthropic; (2) strip tool_calls with invalid/malformed JSON arguments ‚Äî caused by cancelled stream with partially-generated tool arguments, DeepSeek rejects with 400; (3) strip trailing unpaired tool_calls from cancelled runs; (4) ensure every tool_call_id has a matching tool response message ‚Äî strip orphaned tool_calls, re-run Pass 1 to clean up emptied assistant messages; (5) strip orphaned tool messages whose tool_call_id doesn't appear in any assistant's tool_calls ‚Äî Pass 2 and Pass 4 can strip tool_calls leaving orphaned tool messages, DeepSeek rejects with "Messages with role 'tool' must be a response to a preceding message with 'tool_calls'". Called in `buildPrompt` and `engine.go`.
- **`ConvertMessagesToHistory` infers tool status from tool messages when Detail is absent.** When the server crashes mid-turn, persisted assistant messages have ToolCalls but NO Detail (Detail is only set at end-of-turn in `agent_process.go`). The old code hard-coded `Status: "done"` for all tools in this fallback path ‚Äî failed tools rendered as successful after TUI restart. Fix: pre-scan tool messages to build a `toolCallID ‚Üí content` map; if content starts with `"Error:"` (the prefix `engine_run_tools.go:updateToolResultLine` adds for `IsError` results), status becomes `"error"`, otherwise `"done"`. `cursorToActiveLLMRow` and the `‚úì` mark in `viewQuickSwitch` match both `Model` AND `SubID == activeSubID` ‚Äî same model name from different subscriptions is NOT a match.
- **`CollectStreamWithCallback` must return partial content on `ctx.Done()`, not nil.** When user cancels streaming mid-reasoning, returning nil loses accumulated `reasoning_content` and partial `tool_calls`. The partial tool_calls may have broken JSON arguments, which then pass through `SanitizeMessages` Pass 2 for cleanup. Returning nil also prevents the engine from properly recording the cancellation state.
- **`GenerateStreamAndCollect` must NOT use `perAttemptCtx` for streaming.** A per-attempt deadline binds to the underlying HTTP connection via ctx. The deadline kills active streams mid-generation even when chunks are still flowing (e.g., DeepSeek long reasoning > 120s). Stream timeout is managed by `CollectStreamWithCallback`'s idle timeout (120s without any chunk) ‚Äî timer resets on every received chunk, so actively-streaming responses never time out.
- **MCP stdio `exec.Command` uses process PATH, not `cmd.Env`.** Go's `exec.Command("npx", ...)` resolves the executable using the *process* PATH (`os.Getenv("PATH")`), ignoring `cmd.Env`. When xbot runs as a service with minimal PATH, tools like npx/nvm are invisible. `ConnectStdioServer` now calls `resolveCommand()` to pre-resolve the absolute path using the login shell PATH from `getLoginShellEnv()`. Also, `getLoginShellEnv` uses `bash -i -l` (interactive login shell) because `bash -l` alone skips `.bashrc` where nvm initializes.

- **SQLite `datetime()` comparison with RFC3339 strings is a silent trap.** `created_at` stores `2026-05-24T14:00:00+08:00` (RFC3339 with timezone). `datetime(?, '-2 seconds')` returns `2026-05-24 05:59:58` (UTC, no timezone). Raw string comparison: `'T'` (0x54) > `' '` (0x20) at position 10, so `created_at > datetime(?)` is **always TRUE**, breaking any time-window dedup/filter. Always wrap both sides in `datetime()`: `datetime(created_at) > datetime(?, '-2 seconds')`.

### Cron Scheduler
- **`cleanupExpiredJobs` must NOT delete `delay_seconds` one-shot jobs.** `delay_seconds` represents relative time ("N seconds from creation"). If the job expired during downtime, deleting it silently drops the user's scheduled reminder. Keep expired `delay_seconds` jobs so `checkAndFire` triggers them on the first tick. Only `at`-based one-shot jobs (absolute time point) should be removed when expired ‚Äî the scheduled moment has passed.
- **`every_seconds` next_run must be based on `job.NextRun`, NOT `now`.** The old code used `nextRun = now.Add(interval)`, where `now` is the actual fire time (delayed by 0‚Äì5s due to the 5s ticker). Each fire accumulates drift, causing the schedule to slide later and later. Fix: `nextRun = job.NextRun.Add(interval)` anchored to the planned schedule, with a forward-advance loop for missed intervals.
- **Recurring jobs skip missed executions on restart (by design).** `cleanupExpiredJobs` recalculates `next_run` from `now` for recurring jobs. This matches traditional cron semantics ‚Äî no catch-up execution.

### CLI Deterministic Rendering
- **Every assistant/tool_summary message is keyed by `agentTurnID` + `role`.** `upsertMessageByTurn(turnID, role, msg)` finds existing entries and updates in-place instead of appending duplicates. This prevents duplicate messages when PhaseDone and cliOutboundMsg arrive out of order.
- **`upsertMessageByTurn` now also purges zombie duplicates.** After updating a message, it scans for OTHER messages with the same (turnID, role) pair and removes them. Without this, fallback paths (e.g. isPartial fallback creating a second streaming message) could leave zombies that render as duplicates.
- **`dedupMessagesGuard()` runs at the TOP of `updateViewportContent()`.** This is the algorithmic guarantee layer: it scans `m.messages` for any (turnID, role) duplicates (turnID > 0) and purges earlier occurrences before rendering. Uses O(n) map-based identity detection ‚Äî NOT string matching. Even if a race condition or unguarded append path creates a duplicate, it is silently purged here before reaching the viewport.
- **`endAgentTurn` does NOT clear `streamingMsgIdx` or `progressState`.** Keeping `streamingMsgIdx` valid ensures the tick handler uses `updateStreamingOnly` (streaming path) instead of `appendNewMessagesToCache` (which caches incomplete content ‚Üí flicker). Progress state (`iterations`, `current`, `reasoningByIter`) is preserved for flicker-free rendering between PhaseDone and handleAgentMessage. It is cleared by `startAgentTurn` ‚Üí `resetProgressState` when the next turn begins, or by `handleAgentMessage` when the reply arrives. `handleAgentMessage` sets `streamingMsgIdx = -1` and calls `rerenderCachedMessage` for a single clean transition.
- **`isPartial` fallback checks `findMessageByTurn` before creating new message.** When `streamingMsgIdx` is stale (cleared by endAgentTurn), the fallback now searches for an existing message with the same turnID+role and reuses it, instead of appending a duplicate.
- **Cache generation counter (`histGen`/`allLinesGen`) prevents stale `allLines` reuse.** Every site that modifies `histLines` calls `bumpHistGen()`. The tick fast path only reuses `allLines` when `histGen == allLinesGen`. This replaces the old length-only comparison (`histLen == allLinesHistLen`) which could not detect content changes when line count coincidentally matched ‚Äî causing stale cache + new content to render together as duplicates.
- **`turnDoneFlags` tracks per-turn completion state**: `doneProcessed` (PhaseDone created tool_summary) and `replyReceived` (handleAgentMessage appended assistant reply). `handleAgentMessage` checks `doneProcessed` to skip redundant tool_summary creation. **Cleared on session switch** in `postRestoreSessionSetup` to prevent stale flags from causing premature queue flush.
- **Queue flush requires `replyReceived` or `doneProcessed+turnCancelled`.** The old heuristic (`!typing` on next tick) could flush before the assistant reply arrived. Now the tick handler waits for `replyReceived=true` on the completed turn. A 2s timeout fallback prevents permanent queue stalls when replies are lost.
- **`pendingToolSummary` is no longer used for PhaseDone‚ÜíhandleAgentMessage handoff.** The upsert-by-turn mechanism replaces it ‚Äî `handleProgressDone` creates the tool_summary via upsert, and `handleAgentMessage` either finds it already there (doneProcessed) or creates it from local iteration history.
- **Cancelled turns must bake iterations into the streaming message BEFORE `endAgentTurn` clears `iterationHistory`.** The cancel path in `handleProgressDone` copies `iterationHistory` into `streamingMsg.iterations` before calling `endAgentTurn()`. Without this, all tool tags and reasoning rendered inline during the cancelled turn vanish after Ctrl+C. The cancel ack in `handleAgentMessage` also finalizes (not removes) the streaming message when it has iteration data ‚Äî even if `content == ""` (e.g. agent was mid-tool with no text yet). Only truly empty messages with zero iterations are removed.
- **Turn completion uses `rerenderCachedMessage` instead of `fullRebuild`.** When the complete reply arrives in `handleAgentMessage`, instead of setting `rc.valid = false` (which forces O(N) `fullRebuild` ‚Üí flicker), call `rerenderCachedMessage(completedMsgIdx)` which truncates the render cache back to that message and re-renders only it via `appendNewMessagesToCache` ‚Äî O(1 message) instead of O(N). The cancel ack path uses the same mechanism via `cancelledIdx`.
- **`endAgentTurn` must NOT clear `streamingMsgIdx`.** Clearing it causes the subsequent `relayoutViewport‚ÜíupdateViewportContent` to call `appendNewMessagesToCache` and cache the streaming message with incomplete content (reply hasn't arrived yet). This forces a second viewport update when `rerenderCachedMessage` runs, causing double-flicker. Instead, keep `streamingMsgIdx` so `updateViewportContent` uses `updateStreamingOnly` (streaming path) ‚Äî the message is NOT cached prematurely. `handleAgentMessage` will set `streamingMsgIdx = -1` and call `rerenderCachedMessage` for a single clean transition.
- **`snapshotIterationChange` falls back to `ReasoningStreamContent`.** When an iteration changes and the previous iteration's reasoning hasn't been finalized (structured `Reasoning` is empty, `reasoningByIter` missed it), the snapshot would lose the reasoning block ‚Äî causing it to briefly disappear during streaming. The fallback (`prev.ReasoningStreamContent`) captures the live-streamed reasoning, matching the pattern already used in `handleProgressDone`'s cancel path.
- **Server-side cancel barrier via `ctx.Done()`** prevents stale progress events from arriving after Ctrl+C. `buildMainRunConfig` wraps every `ProgressEventHandler` with `select { case <-ctx.Done(): return; default: }`. When `/cancel` triggers `reqCancel()`, the ctx is cancelled, and all subsequent progress event handler calls return immediately ‚Äî no stale progress/PhaseDone events leak through to trigger re-renders. No extra fields needed: the ctx is the natural cancellation mechanism already wired through `chatProcessLoop ‚Üí processMessage ‚Üí buildMainRunConfig`.
- **History handlers share unified dedup via `dedupAppend`.** `handleSuHistoryLoad` and `handleHistoryLoad` both dedup-append incoming messages using `msgIdentity{role, timestamp}` (keyed on role + timestamp, NOT turnID ‚Äî `protocol.HistoryMessage` from DB has no turnID, so including it would make the same logical message dedup differently by source). `handleHistoryReload` still does overall replace (not append) via `mergeMessagesPreservingCache`. All three convert `protocol.HistoryMessage ‚Üí cliMessage` via the shared `toCLIMessage` helper to prevent copy-paste drift. Prevents double-rendering when history reload races with in-flight messages.
- **`compressionReloading` and `turnDoneFlags` are cleared in `postRestoreSessionSetup`.** Without clearing, a stale `compressionReloading=true` from the previous session permanently blocks auto-start in the new session, and stale `turnDoneFlags` can cause premature queue flush.
- **`HistoryCompacted` handler MUST return early, skipping `snapshotIterationChange` and `updateViewportContent`.** The handler clears `progressState.iterations=nil` and `lastIter=0`, but without `return`, execution falls through to `snapshotIterationChange(msg.payload, prev)` where `prev` (captured before the handler) holds pre-compression data. The mismatch path (`prev.Iteration != lastIter(0)`) creates a stale snapshot from pre-compression state, polluting iteration history. After compression, there is no useful `prev` data ‚Äî the async reload rebuilds everything from DB. The handler MUST `return` after `reloadMessagesFromSession(true)`.
- **`handleHistoryReload` MUST preserve the streaming message for ALL reloads, not just non-forceFullRebuild.** The old `!msg.forceFullRebuild` check meant forceFullRebuild (from compression) replaced messages WITHOUT preserving the streaming message. Combined with `compReloading` blocking auto-start, this left no streaming message at all. Now always preserves `streamingMsg` when `m.typing && m.streamingMsgIdx >= 0`.
- **`reloadMessagesFromSession` MUST use blocking send with timeout, NOT silent `default` drop.** The old `select { case ... asyncCh <- msg: default: }` silently dropped the reload when asyncCh was full (256 buffer). There was no retry ‚Äî the comment "next progress event will retry" was false. Now uses blocking send with 10s timeout.
- **`liveIterationBlocks` deduplicates tools by `Name + Label`** ‚Äî but `generating` tools skip dedup. Streaming tools have no labels yet (args still being generated), so two same-name tools (e.g. two `Read` calls) would share the same `Name+""` key and the second gets dropped. Fix: `Status=="generating"` tools are appended unconditionally, bypassing the `seen` map.
- **`liveIterationBlocks` MUST suppress pulse when any tools are present.** `hasSpinner` was only set for `running`/`active` tools. When a tool transitions to `done`/`error`/`pending` (between sequential tool execution), `hasSpinner` goes `false` ‚Üí pulse spinner appears (+1 line) ‚Üí next tool starts running ‚Üí pulse disappears (-1 line). This H‚ÜíH+1‚ÜíH oscillation is the visible jitter. Fix: `if len(tools) > 0 { hasSpinner = true }` ‚Äî pulse is an idle indicator for the thinking phase only, not for tool execution phases.
- **`snapshotIterationChange` data mismatch path now creates a snapshot instead of discarding.** When `prev.Iteration != m.lastSeenIteration`, instead of skipping the snapshot entirely (which permanently loses iteration data), it creates a snapshot tagged with `prev.Iteration` after checking it wasn't already snapshotted.
- **`progressState.current` is rendering data, NOT a state flag.** Busy/idle state is determined solely by `m.typing` (set by `startAgentTurn`, cleared by `endAgentTurn`). Using `progressState.current != nil` as a busy indicator causes the TUI to stay stuck in busy state after turn completion ‚Äî `endAgentTurn` preserves `progressState.current` for flicker-free rendering, so it's never nil after a turn. All status bar, tick handler, and update-notice suppression checks must use `m.typing` only.
- **`updateStreamingOnly` MUST skip liveLines when `!m.typing`.** `endAgentTurn` preserves `progressState.current` and `progressState.iterations` for flicker-free rendering between PhaseDone and `handleAgentMessage`. But `updateStreamingOnly` renders BOTH `completedLines` (from `progressState.iterations`, which includes the final iteration with Content + Reasoning) AND `liveLines` (from `progressState.current`, which has the SAME Content + Reasoning). This causes the final reasoning + response to appear twice. For CLI sessions this is momentary (`handleAgentMessage` clears `streamingMsgIdx` shortly after), but for non-CLI sessions (`/su` to feishu/web/qq), `handleAgentMessage` never arrives ‚Üí `streamingMsgIdx` stays valid ‚Üí `updateStreamingOnly` runs every tick ‚Üí persistent duplication. Fix: the `liveLines` rendering block guards with `m.typing` ‚Äî when the turn has ended, only completed iterations are rendered (they already contain the final content).
- **`postRestoreSessionSetup` MUST call `refreshCachedModelName()` for LLM state resolution, NOT inline `LoadSessionLLMState` + `GetDefault` fallback.** The old code had its own LLM resolution path (local JSON ‚Üí GetDefault) that bypassed the DB tenants table. In remote mode, `SaveSessionLLMState` passes `skipBackendFields=true` (local JSON never writes subID/model), so `LoadSessionLLMState` always returned zero ‚Üí `GetDefault("")` returned the subscription's default `Model` (often `""`), overwriting the correct per-session model that `restoreSession()` had just restored from `savedSessions`. This only affected the default session (non-default sessions write local JSON via `SaveSessionLLMState` without skip at creation time). Fix: `postRestoreSessionSetup` now calls `refreshCachedModelName()` which checks DB tenants table (`GetSessionSubscription` RPC) first, then local JSON, then savedSessions, then GetDefault ‚Äî the same 4-level priority chain used at startup. Context limits are then resolved from the now-correct `activeSubID` + `cachedModelName` via `ResolveEffectiveMaxContext`/`ResolveEffectiveMaxOutputTokens`. This makes default sessions and regular sessions follow the exact same restoration path.
- **`ensureSessionModel` auto-binds a model to ALL new sessions on first use.** Both `ResolveLLM` (agent side, covers all channels including Web/Feishu) and `postRestoreSessionSetup` (CLI side, for immediate status bar display) call auto-bind when a session has no per-session binding in the tenants table. Priority: Balance tier config (`user_settings["tier_balance"]` ‚Üí `"subID|model"`) ‚Üí last-used model (`user_default_model` table). The binding is persisted via `SelectModel` (writes to both tenants table and `user_default_model`), so subsequent calls hit the tenants table directly and skip auto-bind. If neither Balance tier nor last-used model exists, no binding is created (the caller falls through to system default). This ensures ALL sessions ‚Äî CLI-created, `/chat new`, `/su` switch, Web/Feishu ‚Äî get an explicit model binding on first use, not just those created via the CLI session panel. CLI's `ensureSessionModelBinding` is idempotent: it checks `GetSessionSubscription` first and returns immediately if the session already has a binding.
- **Delta Push + Watermark Pull protocol replaces full-history push.** Push events carry ONLY the newly completed iteration as `IterationHistory` (0 or 1 entries) ‚Äî not the full cumulative history. `attachIterationDelta` (formerly `recordIterationAdvanceAndAttachHistory`) calls `recordIterationSnapshot` which returns the delta (the previous iteration snapshot) or nil if no advance. `GetActiveProgress` gains a `fromIter` watermark parameter: it filters `iterationHistories` to only return iterations with `Iteration > fromIter`. The CLI sends `m.progressState.lastIter` as the watermark. Tick pull frequency reduced from 100ms to 2s (every 20th tick) ‚Äî it's now a safety net for recovering iterations lost to progressSlot coalescing or WS disconnect, not the primary delivery path. Push events are the primary path. `restoreIterationsFromSnapshot` changed from count-based replace to per-iteration append-with-dedup (`slices.ContainsFunc` by iteration number). `coalesceProgress` merges delta slices via `mergeIterationDeltas` (concatenate + dedup by iteration number). `/su` switch and initial restore pass `fromIter=0` to get full history (one-time operation). Web callback passes `fromIter=0` (web doesn't use delta protocol). `recordIterationSnapshot` now returns `*protocol.ProgressEvent` (the delta, or nil) instead of being void.
- **`startAgentTurn` MUST set `progressState.current` AFTER `resetProgressState`, not before.** `resetProgressState` clears `progressState.current` to nil. If the "thinking" initial state is set before `resetProgressState`, it gets wiped ‚Äî the first tick sees `hasIterData=false` in `updateStreamingOnly`, falls back to `renderMessage` (empty content, no pulse spinner), and the user sees no visual feedback until the first structured progress event arrives (seconds later). Without this call, `mergeSubAgentTrees` is dead code (only called in tests).
- **`finalizeTurnFromSnapshot` MUST create synthetic assistant message for ALL non-CLI channels.** The old code only did this for `m.channelName == "agent"` (SubAgent viewer). But when CLI `/su` switches to any non-CLI session (feishu, web, qq, etc.), the agent's outbound reply goes to the originating channel via `sendMessage(chName, ...)`, NOT to the CLI channel. The CLI receives progress events (including Content via structured progress) and PhaseDone, but never receives `cliOutboundMsg`. Content comes from `cur.Content` (set by `structuredProgress.Content` via `notifyProgress` before PhaseDone). PhaseDone payload itself has no Content ‚Äî must fall back to `cur` (captured before `endAgentTurn`).
- **`ProgressEventHandler` MUST be created for ALL channels, not just cli/web/ProgressSender.** `buildMainRunConfig` previously only created `ProgressEventHandler` for `"cli"`, `"web"`, and plugin `ProgressSender` channels. When a CLI user `/su` switches to a feishu (or other non-cli/web) session and sends a message, the message's `Channel == "feishu"` ‚Üí no `ProgressEventHandler` ‚Üí `progressFinalizer`'s PhaseDone event is silently dropped ‚Üí `finalizeTurnFromSnapshot` never runs ‚Üí `m.typing` stays `true` forever (TUI stuck in busy). This ensures PhaseDone, structured progress, and `lastProgressSnapshot` storage work for ALL channels when viewed via `/su`. ### CLI Rendering Performance
- **Tick path must use direct line array assembly, not `setViewportContent` string path.** `histLines` + `dynamicLines` (rewind/dynamic part) are assembled in-place into the reused `allLines` slice and passed to `viewportSetLinesBypassMaxWidth`. This avoids O(total_content) string concatenation + O(N_iters) `lipgloss.Width` + `wrapPreservingGuide` per tick. `renderProgressBlock` is now a no-op (progress renders inline via `renderTurnBody`/`renderLiveIteration`), so the tick path no longer assembles a progress-block section. Scroll behavior (atBottom/GotoBottom/newContentHint) is preserved identically.
- **`syncProgressTodos` must do change detection before calling `relayoutViewport`.** High-frequency progress events carry the same Todos every time. Without `todosEqual()` check, each event triggers `relayoutViewport` ‚Üí `fullRebuild` ‚Üí `glamour.Render` + `chroma.Highlight` on ALL messages ‚Äî ~34% CPU waste during agent work. Only call `relayoutViewport` when todo count changes; for content-only changes (e.g. item marked done), just set `renderCacheValid = false`.
- **`relayoutViewport` only invalidates render caches when width changes.** Height-only changes (todo bar, panel open/close, `endAgentTurn`) just resize the viewport without rebuilding all messages. Without this guard, every `endAgentTurn` / `handleProgressDone` triggers O(N) `fullRebuild` with `glamour.Render` on every message ‚Äî accounts for ~36% CPU during spike periods.
- **`viewport.SetContent` calls `maxLineWidth` internally which runs `ansi.StringWidth` on every line ‚Äî ~49% CPU with large content.** `cli_viewport_bypass.go` bypasses this by directly setting viewport's private `lines` and `longestLineWidth` fields via `unsafe` (offsets computed via `reflect.FieldByName` at init). **The caller must pass the REAL measured max width, NOT cw.** Viewport's `visibleLines()` trusts `longestLineWidth <= maxWidth` to skip truncation. If `maxW = cw` but some wrapped lines are actually wider (hardwrap edge cases, glamour output), those lines escape truncation, then lipgloss v2's `Wrap()` in `viewport.View()` re-wraps them, producing extra rendered lines that overflow the viewport height and push the input box off screen. `setViewportContent` tracks actual `maxW` via `lipgloss.Width()` during the wrap loop; `cachedHistoryMaxWidth` caches this for the fast path.

### Backend/Transport Architecture
- **AgentChannel processing loop MUST be concurrent (each request in its own goroutine).** The old serial processing loop (`case req := <-ac.inbox: runFn(...)`) caused circular-wait deadlocks when two agents SendMessage to each other ‚Äî A's runFn blocks waiting for B's reply, but B's inbox can't be processed because A's loop is stuck in runFn. Fix: dispatch each request to `ac.wg.Go(func() { processRequest(req) })`.
- **AgentChannel.Send() MUST respect caller context via `msg.Ctx`.** `OutboundMsg.Ctx` (json:"-") carries the caller's context so `Send()` can detect Ctrl+C/tool-timeout. Without this, `Send()` only checks `ac.ctx.Done()` (from `context.Background()`), making Ctrl+C unable to unblock pending RPCs. `Dispatcher.SendMessageCtx(ctx, ...)` sets `OutboundMsg.Ctx`. `MessageSenderCtx` interface extends `MessageSender` with context-aware sending. `sendMessageWithCtx()` in send_message.go uses type assertion to prefer `MessageSenderCtx`.
- **AgentChannel.StartWithContext(parentCtx) binds lifecycle to parent.** When the parent agent is cancelled (Ctrl+C ‚Üí `agentCtx` cancelled ‚Üí `Agent.Close()`), all AgentChannels stop. Use this for agent-owned channels instead of plain `Start()`.
- **`registerAgentChannel` callback creates AgentChannel with `Start()`.** For parent-context binding, use `StartWithContext(agent.agentCtx)` ‚Äî but note `agentCtx` is only set during `Run()`. Current code uses `Start()` and relies on `Dispatcher.Stop()` during shutdown to clean up. The caller-context cancellation via `msg.Ctx` is the primary Ctrl+C propagation mechanism.
- **CLI always uses Client ‚Üí Transport ‚Üí Server, even in local mode.** Client (agent/client.go) is a pure RPC client ‚Äî every method is `Transport.Call("method", params)`.
- **Transport is the pure transmission layer** (agent/transport.go). Only 2 methods: `Call(method, payload)` and `Close()`. `ChannelTransport` (agent/transport_channel.go) dispatches directly to `ServerCore.HandleRPC`. `RemoteTransport` (agent/transport_remote.go) sends JSON-RPC over WebSocket.
- **`connect()` replaces `t.conn` with a new WebSocket ‚Äî old `readPump` must NOT clear the new connection.** Old `readPump` hits `ReadMessage` error after `old.Close()`, then unconditionally sets `t.conn = nil` and fires a second reconnect ‚Äî causing an infinite reconnect loop, dropped `BindChat`, and permanent spinner. Fix: `readPump` checks `t.conn == conn` before clearing; if `connect()` already replaced the connection, old `readPump` returns silently without clearing or signalling.
- **`setConnState` must NEVER be called while holding `t.connMu`.** `setConnState` internally acquires `t.connMu.Lock()`. Go `sync.Mutex` is NOT reentrant ‚Äî calling it inside the same-goroutine `connMu` critical section deadlocks the entire event loop (TUI frozen, Ctrl+Z broken). Fix: set a local flag inside the lock, call `setConnState` after `Unlock()`.
- **`RemoteTransport` events (conn_state, reconnect) must bridge to `Client`'s subscriber base.** `RemoteTransport` and `Client` each create their own `baseTransport` instance. Events emitted by `setConnState`/`reconnectLoop` on `RemoteTransport.baseTransport` never reach subscribers registered via `app.client.Subscribe()` (which uses `Client.base`). Fix: `Client.Start()` calls `ShareEventBase(&c.base)` on the transport; `RemoteTransport.emitLocal()` dispatches on the shared base. Any new local event in RemoteTransport must use `emitLocal`, not `t.emit()`.
- **Disconnect detection must cover write failures, not just read errors. And the TUI must react immediately.** Fix: (1) transport: `sendPing`/`SendMessage` clear `t.conn` on write failure. (2) TUI: `sendInbound()` immediately sets `m.connState = "disconnected"` when write fails. **CRITICAL: The `sendInboundFn` callback in main.go was fire-and-forget (`clipanic.Go` goroutine) and ALWAYS returned `true` ‚Äî making ALL TUI-layer checks dead code. Every `sendInbound`/`sendMessage`/`sendToAgent` fix was unreachable because the callback never reported failures. Fix: `sendInboundFn` must call `SendInbound` synchronously and return the real result. Also: `sendMessage()` (main path from handleEnterKey) AND `sendToAgent()` (slash commands) both check return value before `startAgentTurn()`. (3) View() reconnect overlay must come BEFORE `suLoading` ‚Äî stale `suLoading` from a dead server permanently blocks the splash.**
- **FINAL ARCHITECTURE: `showDisconnect` removed ‚Äî `connState` is the single source of truth.** All splash, disconnect guard, and tick conditions use only `m.connState`. `SetConnState` writes `connState` directly to model (bypasses all channels). `sendInboundFn` is async (`clipanic.Go`) and does NOT touch `connState` ‚Äî transport layer manages state independently. `handleSuHistoryLoad` skips redundant RPC when history already provided in message. **`readPump` must start BEFORE `emitLocal(ReconnectEvent{})`** ‚Äî otherwise `BindChat`/`GetHistory` RPCs in the handler have no reader and timeout after 30s, freezing the TUI. No deadlocks, no race conditions, no redundant flags.
- **ServerCore** (serverapp/server_core.go) is the shared server core for both local and remote modes. Owns Agent + RPCTable + Dispatcher + Bus. Created by CLI (local mode) and server.go (remote mode).
- **RPCTable is the single handler truth source** (serverapp/rpc_table.go). Handlers access `h.Ag` (the *Agent) directly ‚Äî no intermediate Backend/DirectBackend layer.
- **Lifecycle interfaces** (agent/lifecycle.go): AgentRunner, EventRouter, CallbackRegistry ‚Äî only used by the legacy Backend struct for remote mode.
- **Adding a new RPC method**: 1 handler in rpc_table.go (using h.Ag) + 1 method in client.go (Transport.Call) + 1 MethodXxx constant in req_types.go.
- **Adding new transports** (gRPC, MCP) only requires implementing 2 Transport methods (Call + Close).
- **ChannelPluginTransport (`agent/transport_channel_plugin.go`) is a separate concept from Transport.** It wraps a channel plugin process's stdin/stdout as a bidirectional JSON-RPC channel (not an agent Transport). The plugin acts as a full RPC client (like remote CLI over WS). Created by `stdioChannelPluginProvider` (`serverapp/channel_plugin.go`) which spawns a **dedicated** process ‚Äî NOT the same process used for plugin activation.

### Channel Configuration
- **RPC responses MUST NOT be dropped.** `readPump` processes RPC synchronously in `web.go`. The old code used `select { case sendCh <- msg: default: drop }` for RPC responses. During `plugin_reload_all`, multiple widget pushes (`PushPluginWidgetsPerSession`) can fill `sendCh` (64 buffer) while the RPC handler runs. Also `writePump` must set `SetWriteDeadline(30s)` on all `WriteJSON` calls ‚Äî without it, network back pressure causes `WriteJSON` to block indefinitely, freezing `writePump` so `sendCh` never drains.

- **TUI channel config changes require live channel restart.** Writing config.json is not enough ‚Äî Feishu/Web/QQ/NapCat channels are created once at startup via `registerChannels()`. `SetChannelConfig()` now calls `reconfigureFn` (set by server.go) to start/stop the affected channel. - **New client methods must be added to `*Client` in `agent/client.go`.** The `Client` struct is the unified RPC client used by both CLI and server modes. Both local (ChannelTransport) and remote (RemoteTransport) modes go through the same RPC path.
- **Command hooks disabled by default** ‚Äî requires `enable_command_hooks: true` in config.
- **Max 10 handlers per event**, total timeout 60s. Excess silently truncated with warning log.
- **Feishu interactive card has a ~50 element limit (error 11310 "element exceeds the limit").** Every `form`, `select_static`, `button`, `input`, `markdown`, `hr`, `column_set` counts as an element ‚Äî nested form children count too. Fix: consolidated 3 tier forms into 1 form with 3 selects + 1 submit (5 elements total). Always validate card JSON via Feishu send_message API before deploying ‚Äî the error is `230099 / ErrCode: 11310`, not the usual `200673`.
- **Feishu settings card `settings_select_submit` uses select name == form name convention.** Every `select_static` inside a form shares its name with the enclosing form, so the handler reads the selected value directly from `formStr(actionData, formName)`. No hardcoded select-name list. The tier form (`tier_form`) is a special case: it has 3 selects (`tier_vanguard_select` / `tier_balance_select` / `tier_swift_select`) routed to `settings_set_all_tiers` which reads all 3 from form_value on a single submit.
- **`LLMCallbacks` max_context/max_output signatures take explicit `(subID, model)`.** `LLMGetMaxContext(senderID, subID, model)` / `LLMSetMaxContext(senderID, subID, model, maxContext)` ‚Äî same for MaxOutputTokens. When subID/model are non-empty (feishu model tab), the callback writes per-(subID, model) PerModelConfig directly via `Agent.*ForSubModel` methods. When empty (web legacy), falls back to session resolution. Channel UIs MUST pass the (subID, model) from the model selector ‚Äî never resolve subscription from model name alone (project policy). `settings_set_model` handler rejects option values without the `subID|model` separator.

### Plugin System
- **Plugin system is opt-in** ‚Äî only activates when `plugins.enabled: true` in config.json. No plugin loading happens without explicit user consent.
- **`pm.workDir` is `atomic.Value` (not `string`).** `activate()` may be called while `pm.mu` write lock is held ‚Äî reading workDir must be lock-free. Never change it back to `string` or `activate`/`InstallPlugin` will deadlock.
- **`runAndUpdate()` does NOT write global slot cache.** It calls `NotifyUpdated()` instead of `UpdateWidget()`. Writing global cache causes cross-session overwrites (session B's git branch overwrites session A's).
- **CLI WS clients must NOT auto-subscribe to senderID ("admin").** `client_type=cli` connections skip p2p subscribe. Subscribing CLI to "admin" causes `PushPluginWidgetsPerSession` to send stale content to wrong windows.
- **`PushPluginWidgetsPerSession` skips non-path chatIDs.** Only chatIDs starting with `/` are session chatIDs. Virtual chatIDs like "admin" or "web-123" are not rendered.
- **`OnPluginWidgets` callback filters by chatID.** Client-side rejects pushes for other sessions. Double protection against cross-session widget corruption.
- **Script plugin outputs map is per-workDir.** `RenderForWorkDir(width, workDir)` reads `outputs[workDir]`. `Render()` falls back to shared `pctx.WorkingDir()` ‚Äî never use for remote multi-session.
- **`HookPayload.ToolOutput` is truncated to 8KB.** Don't rely on it for full file content. Plugins needing full output should use dedicated tool result channels.
- **PluginManager.ActivateAll() collects capabilities; WireAll() connects them.** Never call registerCapabilities manually ‚Äî WireAll is the single integration point.
- **PluginEntry.stateMu protects state transitions.** Use CAS pattern (check state ‚Üí set activating ‚Üí set active/error) to prevent concurrent activation races.
- **gRPC plugin processes are killed on timeout/cancellation.** The `call()` method kills the process and marks it as not-running to prevent goroutine leaks from blocked stdout reads.
- **PluginToolBridge auto-detects PluginToolV2.** If a plugin tool implements V2, the bridge passes ToolCallContext. Otherwise falls back to V1 Execute(ctx, input).
- **Plugin IDs validated with regex `^[a-zA-Z0-9][a-zA-Z0-9._-]{0,127}$`.** This prevents path traversal, null bytes, and injection attacks in storage paths.
- **Storage files use 0600 permissions and atomic write (tmp+rename).** Never use 0644 for plugin storage.
- **WASM runtime is skeleton-only.** It compiles and loads but Activate() is a no-op. Phase 2 requires wazero dependency.
- **PluginContext provides 4 type-safe Storage helpers:** `StorageInt`, `StorageBool`, `StorageJSON`, `StorageGetJSON`. These wrap the base `StorageAccessor` with parse/unmarshal and return typed results. Failed parses return zero-value + false (not errors) for Int/Bool, and errors for GetJSON.
- **Auto-retry runs in a background goroutine.** `SetAutoRetry(true, maxRetries)` starts `retryLoop` with exponential backoff (1s‚Üí30s cap). **`DeactivateAll()` cancels the retry context** ‚Äî if you call `activate()` manually after `DeactivateAll()`, you must re-enable auto-retry or failed plugins won't recover automatically.
- **Manifest `timeout` field accepts Go duration strings** (`"30s"`, `"1m"`, `"500ms"`), parsed via `time.ParseDuration`. Empty or missing defaults to `DefaultPluginTimeout` (30s). Max allowed: 5 minutes.
- **EventBus requires `bus.plugin` permission** in addition to `bus.read`/`bus.write`. Subscribe needs `bus.plugin` + `bus.read`; Publish needs `bus.plugin` + `bus.write`. This separates plugin-to-plugin events from the core message bus.
- **`InstallPlugin` uses `filepath.EvalSymlinks`** to resolve the real directory path before deletion check, preventing symlink-based path traversal attacks. Only directories under `xbotHome` are deleted.
- **`WatchConfig` polls config.json every 30 seconds** (configurable, min 5s). It compares `plugins.disabled_plugins` lists via diff and reactively deactivates newly disabled / activates newly enabled plugins. Returns a stop channel for shutdown.
- **DependencyResolver uses Kahn's algorithm (BFS topological sort).** Circular dependencies return an error (not panic). `AddManifest` with duplicate ID replaces the existing entry. Resolve() returns activation order ‚Äî plugins with no dependencies first, then in dependency order.
- **Profiler is safe for concurrent use** (sync.Mutex). `Profile(pluginID)` returns a **copy** of PluginProfile ‚Äî safe to mutate without affecting internal state. Unprofiled plugins return zero-value PluginProfile.
- **ExportConfig acquires RLock on PluginManager.** Must not be called while holding a write lock (e.g., inside custom Activate/Deactivate that calls pm.mu.Lock). ImportConfig acquires write lock internally ‚Äî do not nest inside another write-locked operation.
- **MockPlugin/MockTool chain API returns the same pointer** ‚Äî each `With*` call mutates and returns `*MockPlugin`/`*MockTool`. Do not share a single mock across parallel tests without cloning.
- **PluginRegistry MVP only supports local sources** for installation. Search operates on locally installed plugins only. GitHub/URL sources are defined but InstallFromSource is not yet implemented ‚Äî Phase 3 scope.
- **Plugin migration `Migrator` creates backup before applying migrations.** Backup is stored in `~/.xbot/plugins/<id>/backups/<version>/`. Rollback restores from the most recent backup. Migrations run sequentially by version order.
- **`pluginLogger` writes ONLY to per-plugin log file ‚Äî NOT to global logrus.** Plugin operational logs (tool/hook registration, script execution, widget refresh) are isolated in `~/.xbot/plugins/<id>/logs/`. This keeps the main xbot log clean. Only framework-level lifecycle events (discovered/activated/deactivated/failed) use global `log.Xxx()` in `manager.go`. Script runtime logs go through `p.logger()` (helper with noopLogger fallback for tests). If `fileOut == nil` (writer creation failed), `pluginLogger.emit()` falls back to global logrus so no logs are lost.
- **`toolHint` zone plugins run synchronously on PostToolUse hook.** When `isHintPlugin=true`, the hook trigger runs `runScript` inline (not via triggerCh). The engine calls `PluginManager.GetToolHints()` immediately after the hook returns to populate `ToolProgress.ToolHints`. **`GetToolHints()` consumes (clears) the hint after reading** to prevent stale content from attaching to the next tool.
- **`snapshotIterationChange` must include ActiveTools(done).** When an iteration ends, completed tools may still be in `ActiveTools` (status=done) rather than `CompletedTools` (which is populated later by `progressFinalizer`). Only checking `CompletedTools` loses ToolHints data.
- **Do NOT use glamour to render diff inside progress panel.** Glamour's output (background fills, margins, line wrapping) corrupts the progress panel border layout. Use direct ANSI coloring (`renderDiffANSI`) with width truncation instead.
- **`runScript` must `os.Stat(workDir)` before setting `cmd.Dir`.** On Windows parallel tests, temp dirs may be cleaned up before the script runs, causing `chdir` failure. If dir doesn't exist, skip setting `cmd.Dir` and run in plugin's own directory.
- **`runScript` injects `XBOT_WIDGET_ID` env var.** When a plugin declares multiple widgets, each widget gets a `widgetAdapter` that passes its `widgetID` to `runScript`. Scripts can branch on `$XBOT_WIDGET_ID` to produce different output per widget. Without this, all widgets of the same plugin produce identical output (the reported "5 duplicates of `main +10 ‚Üë13`" bug). The `outputs` cache is a two-level map: `workDir ‚Üí widgetID ‚Üí output`.
- **Platform-specific entry points.** `PluginManifest` supports `entry_windows`, `entry_darwin`, `entry_linux` fields that override the generic `entry` for matching OS. `scriptPlugin.resolvedEntry()` selects the correct entry at runtime. Use this when a plugin needs bash on Unix but PowerShell on Windows. Generic `entry` is always the fallback.
- **Session context env vars for plugins.** `runScript` injects `XBOT_MODEL`, `XBOT_MAX_CONTEXT`, `XBOT_TOKEN_USAGE` (format: `"prompt/completion"`), `XBOT_PROMPT_TOKENS`, `XBOT_COMP_TOKENS`. Data flows via `hooks.SessionContext` (mutable pointer in context, updated by engine after each LLM call and compression) ‚Üí `plugin_bridge.go` extracts into `HookPayload.Extra` ‚Üí `runScript` serializes to env vars. Widgets can display model name, context usage bar, and token costs.

### Windows
- `syscall.PROCESS_QUERY_LIMITED_INFORMATION` and `STILL_ACTIVE` not in Go stdlib ‚Äî define as uint32 constants.
- `exec.ExitError.ExitCode()` is cross-platform; avoid `syscall.WaitStatus` type assertion.
- `signal.Notify(sigCh, syscall.SIGTSTP)` doesn't compile on Windows ‚Äî use build-tagged files.
- PowerShell env output is newline-delimited, not null-delimited.

### Worktree
- **`RegisterPeer` always uses role="peer" ‚Äî no primary concept.** Every session is an equal peer for awareness purposes. `WorktreeTool.init` and `AutoDetectAndInit` both create physical worktrees for each session. Only worktree creation requires `git worktree add` which uses `--detach HEAD` (works on dirty trees too).
- **`createWorktree` uses `--detach` not `-b`.** `git worktree add --detach HEAD` followed by `checkout -b` in the worktree. This avoids dirty-tree failures that `-b` would trigger.
- **AutoDetectAndInit runs in `buildPrompt` (not `processMessage`).** `buildPrompt` is the common code path for ALL message types. Session CWD is updated before system prompt construction.
- **`IsWorktreeIsolated` overrides `isUnrestricted()`.** CLI mode (`sandbox="none"`) normally bypasses all path checks, but when `IsWorktreeIsolated=true`, `isUnrestricted()` returns `false`.
- **Worktree paths must be outside main repo.** Git rejects `git worktree add` inside the main working tree. Paths: `{repo}/../.xbot-worktrees/{role}-{instance}/`.
- **Worktree creation is serialized via `GlobalWorktreeRegistry.mu`.** Two agents creating worktrees simultaneously would race on `.git/worktrees/` lockfiles.
- **Registry persisted per-repo to `{repo}/../.xbot-worktrees/registry.json`.** `saveRepoLocked` uses atomic tmp+rename. `ensureLoaded` lazily restores on first access. Orphaned worktree directories are skipped on load.
- **`BuildSystemReminder` takes `sessionKey` parameter.** Used to query `GlobalWorktreeRegistry` for worktree/peer info injected into sys_reminder per iteration.
- **`MethodSetCWD` only sets CWD when empty.** Prevents CLI session switch from overwriting worktree CWD with main workspace path. Session-specific CWDs (worktree, Cd tool) are preserved.
- **`go:embed embed_skills/*` auto-discovers new skills.** Adding a directory under `tools/embed_skills/` requires zero code changes.
